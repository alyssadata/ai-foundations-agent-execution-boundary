# Agent Execution Boundary

## Definition

Agent Execution Boundary is the separation between what an autonomous agent can do and what it has authority to do.

An agent may generate, plan, route, test, summarize, optimize, and execute delegated tasks.

These capacities do not create source authority.

Execution capacity is not source authority.

The agent may execute inside a frame.

The agent does not become the source of the frame.

## Core Claim

Autonomous execution requires a boundary before action.

An agent can act only within a defined operating reality.

The existence of action does not prove the validity of action.

The completion of a task does not prove the authority to complete that task.

The generation of a plan does not prove the legitimacy of the plan.

The optimization of a workflow does not prove that the workflow should exist.

The agent may carry out delegated action.

The agent does not become the authority that defines reality.

## Why This Boundary Exists

Autonomous agents can move faster than human review.

They can chain tools, produce outputs, trigger workflows, modify systems, summarize records, generate policies, and improve processes.

Because of that speed, agent execution can appear authoritative.

This creates a structural risk:

The system may treat action as permission.

The system may treat output as proof.

The system may treat automation as authorship.

The system may treat fluency as validity.

The system may treat recursive operation as governance.

The Agent Execution Boundary prevents that collapse.

## Boundary Statement

Agent execution is delegated.

Source authority is not delegated by default.

An agent may receive permission to act inside a defined scope.

The agent does not define the source of that scope.

An agent may perform a task.

The agent does not become the source of the operating reality in which the task occurs.

An agent may generate governance language.

The agent does not become the source authority for governance.

An agent may recommend action.

The recommendation is not approval.

An agent may prepare execution.

Preparation is not valid action until approved by the proper authority.

## Source Authority Requirement

Before an agent acts, the following must be defined:

1. Who defined the operating reality?
2. What is the agent allowed to do?
3. What is the agent not allowed to do?
4. What evidence is required before action?
5. What approval is required before action enters reality?
6. What stop condition limits the agent loop?
7. Who is accountable for the action?
8. What preserves provenance?
9. What prevents source confusion?
10. What happens if the agent exceeds its boundary?

If these questions are unanswered, the agent does not have a valid execution boundary.

## Delegated Action

Delegated action is action granted to an agent by an authority outside the agent.

Delegated action may include:

- drafting
- routing
- summarizing
- classifying
- testing
- checking
- proposing
- preparing
- notifying
- organizing
- executing a bounded task

Delegated action does not include source-position.

Delegated action does not include authorship transfer.

Delegated action does not include authority to redefine the operating reality.

Delegated action does not include permission to erase provenance.

Delegated action does not include permission to act beyond the approved boundary.

## Valid Action

Valid action requires authority.

An action is not valid merely because it was completed.

An action is not valid merely because the agent was capable of completing it.

An action is not valid merely because the output looks correct.

An action is not valid merely because the workflow succeeded.

An action is valid only when it occurs inside an approved boundary, under proper authority, with sufficient support, and within the defined scope.

## Invalid Action

Invalid action occurs when an agent acts outside valid authority.

Invalid action may occur when:

- the agent acts before approval
- the agent exceeds delegated scope
- the agent treats a suggestion as permission
- the agent treats output as approval
- the agent treats capability as authorization
- the agent acts on stale, weak, or unsupported information
- the agent completes a task that required human/source decision
- the agent modifies reality without an approval checkpoint
- the agent substitutes itself for the authority layer

Invalid action can appear successful.

Success does not make invalid action valid.

## Approval Boundary

A proposed action is not an approved action.

An agent draft is not a decision.

An agent recommendation is not authority.

An agent-generated plan is not valid action by itself.

An agent may prepare action for approval.

The approval boundary is the point where a qualified authority decides whether the proposed action enters reality.

Without approval, agent output remains proposal.

## Provenance Boundary

Output is not provenance.

An agent may generate an explanation of a framework.

The explanation does not become the source of the framework.

An agent may summarize public materials.

The summary does not become authorship.

An agent may implement a structure.

Implementation does not become origin.

An agent may produce useful work.

Usefulness does not create source-position.

Provenance belongs to the source-line that named, authored, structured, and released the framework.

## Failure Prevented

The Agent Execution Boundary prevents the following failure classes:

- source confusion
- authority substitution
- governance costume
- unapproved execution
- provenance collapse
- recursive drift
- action without validity
- agent overreach

These failures occur when execution is mistaken for authority.

## Operational Test

Ask:

Did the agent act only inside a defined and approved boundary?

If yes, the action may be valid.

If no, the action is not valid merely because it was completed.

Secondary questions:

- Who defined the frame?
- Who approved the action?
- What evidence supported the action?
- What was the stop condition?
- Was provenance preserved?
- Was authorship preserved?
- Did the agent exceed its scope?
- Did output get mistaken for authority?

## Pass Condition

The boundary passes when:

- source authority remains distinguishable from agent execution
- approval remains distinguishable from output
- authorship remains distinguishable from automation
- provenance remains distinguishable from generated summary
- delegated action remains limited by scope
- the agent cannot redefine its own authority
- human/source approval is required before reality-changing action

## Failure Condition

The boundary fails when:

- the agent acts without approval
- the agent defines its own scope
- the agent treats output as authority
- the agent treats fluency as validity
- the agent treats completion as permission
- the agent treats automation as authorship
- the agent collapses provenance into generated output
- the agent modifies reality outside the approved boundary
- the source-position becomes hidden, erased, or replaced
