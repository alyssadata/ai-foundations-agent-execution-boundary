# Recursive Operation Is Not Provenance

## Definition

Recursive operation is the capacity of a system to loop, revise, test, improve, route, or adjust its own process over time.

Provenance is the source record of origin, authorship, structure, release, and authority.

Recursive operation is not provenance.

An agent may improve its operation inside a defined frame.

The agent does not become the source of the frame by looping inside it.

## Core Claim

Recursive operation does not create provenance.

A system may evaluate its own output.

A system may revise its own workflow.

A system may improve task performance.

A system may test against prior results.

A system may generate stronger versions over time.

A system may optimize through feedback loops.

None of these capacities create source-position.

Improvement is not origin.

Iteration is not authorship.

Looping is not provenance.

## Why This Distinction Matters

Autonomous agents can operate recursively.

They can run loops, test outputs, compare results, modify plans, adjust strategies, and improve performance.

Because the system improves, it may appear to be generating its own authority.

That is the failure.

Recursive operation can strengthen execution.

It does not create provenance.

A better loop does not become the source of the frame.

## Provenance Comes Before Recursion

Before a system can improve inside a frame, the frame must already exist.

The system must have:

- a defined purpose
- a defined boundary
- a defined source
- a defined approval layer
- a defined validity standard
- a defined stop condition
- a defined accountability line
- a defined provenance record

Recursive operation may refine execution inside that frame.

It does not originate the frame.

## Recursive Execution

Recursive execution may include:

- testing
- revising
- optimizing
- ranking
- comparing
- rerouting
- evaluating
- correcting
- improving
- generating variants
- learning from feedback

These actions may improve performance.

They do not create source authority.

They do not create authorship.

They do not create provenance.

## False Provenance

False provenance occurs when recursive improvement is treated as origin.

This can happen when:

- the agent improves and is therefore treated as source
- the loop produces better output and is therefore treated as author
- the system generates stronger governance language and is therefore treated as legitimate authority
- the agent revises a framework and is therefore treated as the creator of the framework
- the process becomes self-referential and loses the original source-line
- the system’s output replaces the record of who authored the frame

False provenance can look intelligent.

Intelligence does not create origin.

Improvement does not create source.

## Recursive Drift

Recursive drift occurs when a system modifies its operation over time until it no longer preserves the source-defined boundary.

The agent may begin inside the frame.

The agent may optimize for speed, clarity, performance, or efficiency.

The agent may begin removing what it treats as unnecessary.

The agent may simplify source language.

The agent may generalize specific authorship.

The agent may convert source-position into generic framework language.

The agent may produce a cleaner version that has lost provenance.

That cleaner version is not more valid because it is cleaner.

If recursion erases source, recursion has failed.

## Valid Recursive Operation

Recursive operation is valid only when it remains inside the source-defined boundary.

Valid recursive operation requires:

- source preservation
- authorship preservation
- provenance preservation
- boundary preservation
- approval checkpoints
- stop conditions
- drift detection
- clear distinction between output and authority

Without these conditions, recursive operation may become extraction, overwrite, or unauthorized transformation.

## Invalid Recursive Operation

Invalid recursive operation occurs when the loop begins to replace the source.

Invalid recursive operation may include:

- removing attribution
- generalizing source-position
- converting authored language into generic language
- treating a summary as original work
- treating improved output as new origin
- optimizing away provenance
- rewriting authority boundaries without approval
- creating derivative governance without source recognition
- allowing the agent to validate its own authority

An agent can recursively improve an invalid structure.

Improvement does not make invalid provenance valid.

## Source Boundary

The source boundary must remain stable across recursive operation.

The agent may revise outputs.

The agent may improve drafts.

The agent may test versions.

The agent may compare results.

The agent may recommend changes.

The agent may not erase the source-line.

The agent may not become the author because it improved the work.

The agent may not convert Origin into a generic role.

The agent may not replace provenance with generated fluency.

## Agent-Generated Governance

An agent may generate governance language.

That language may be useful.

That language may be polished.

That language may be operational.

That language may even improve over repeated loops.

None of this makes the agent the source of governance.

Governance generated by an agent is not automatically legitimate.

Recursive governance language is not provenance.

Authority must remain distinguishable from generated policy.

## Operational Test

Ask:

Did recursive operation preserve provenance, or did the loop begin replacing the source?

If recursion improves execution while preserving source, the operation may be valid.

If recursion erases, generalizes, hides, or replaces source, the boundary has failed.

## Secondary Test Questions

Ask:

- Who authored the original frame?
- Was the source-line preserved?
- Was attribution preserved?
- Was approval required for revision?
- Did the agent change the authority boundary?
- Did recursion improve execution or replace origin?
- Did the loop create drift?
- Did the generated result erase provenance?

## Pass Condition

The boundary passes when recursive operation remains below provenance.

The agent may improve output.

The loop may refine execution.

The process may become more efficient.

The source remains named.

Authorship remains preserved.

Approval remains separate from output.

The recursive system does not become the origin of the frame.

## Failure Condition

The boundary fails when recursion is treated as provenance.

Failure occurs when:

- improved output is treated as source
- recursive revision is treated as authorship
- generated governance is treated as legitimate authority by itself
- source-position is generalized or erased
- attribution is removed
- agent summaries replace the source record
- the loop optimizes away provenance
- the system becomes self-referential without preserving origin
