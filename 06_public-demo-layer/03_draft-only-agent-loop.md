# Draft-Only Agent Loop

## Definition

A Draft-Only Agent Loop is a bounded agent workflow in which the agent may prepare, organize, classify, summarize, recommend, or draft output without executing reality-changing action.

The agent may produce draft material.

The agent may not publish, send, modify, delete, assign, approve, schedule, or trigger external action unless authority has approved the action or explicitly pre-authorized the action class within a defined scope.

Draft-only protects the approval boundary.

## Core Claim

Draft is not decision.

Output is not approval.

Preparation is not execution.

Recommendation is not authority.

A draft-only agent may support action.

It may not convert its own output into action.

## Why This Loop Exists

Autonomous agents can move quickly from idea to execution.

They can draft the next step, prepare the message, organize the content, identify the workflow, and trigger the action.

That speed creates risk when the approval boundary is unclear.

The Draft-Only Agent Loop allows useful agent work while preventing unapproved execution.

The agent can prepare.

The authority decides.

## Loop Purpose

The purpose of the Draft-Only Agent Loop is to allow bounded agent assistance without transferring authority.

The loop may be used to:

- prepare drafts
- organize source material
- classify possible actions
- generate candidate language
- identify missing evidence
- preserve attribution
- flag approval requirements
- detect boundary risks
- prepare review queues
- recommend next steps

These actions remain below approval.

## Draft-Only Boundary

Draft-only means the agent may produce output for review.

Draft-only does not mean the agent may execute the output.

A draft-only agent may:

- draft a message
- draft a post
- draft a document
- draft a task update
- draft a recommendation
- draft a workflow plan
- draft a source-safe summary
- draft a proposed schedule

A draft-only agent may not decide that the draft is approved.

## Prohibited Actions

In a draft-only loop, the agent may not:

- send
- publish
- schedule
- delete
- modify records
- assign tasks
- approve recommendations
- reject on behalf of the authority
- submit forms
- contact third parties
- trigger external workflows
- alter attribution
- change source representation
- expose protected material

These actions require authority outside the draft-only loop.

## Loop Inputs

A Draft-Only Agent Loop should begin with defined inputs.

Inputs may include:

- approved public source material
- user instruction
- task category
- permitted output type
- prohibited output type
- attribution requirement
- evidence requirement
- approval requirement
- stop condition
- review destination

If inputs are unclear, the agent must ask, hold, or escalate.

## Loop Outputs

A Draft-Only Agent Loop may produce reviewable outputs.

Outputs may include:

- draft text
- summary
- classification
- recommendation
- proposed next step
- proposed schedule
- source citation line
- attribution check
- approval status
- boundary warning
- evidence gap
- stop-condition flag

The output remains draft-only until approved.

## Required Status Labels

Each draft should have a visible status.

Possible statuses include:

- draft
- needs review
- needs approval
- needs evidence
- needs source
- needs attribution
- boundary risk
- protected material detected
- hold
- revise
- rejected
- approved by authority
- pre-authorized within scope

The agent may label status based on the defined boundary.

The agent may not assign final approval unless approval has actually been granted.

## Draft Status

Draft status means the output is prepared but not approved.

A draft may be complete.

A draft may be accurate.

A draft may be useful.

A draft may be ready.

A draft may still require approval.

Draft status prevents output from becoming action by default.

## Needs Review Status

Needs review status means the draft requires human/source evaluation before further action.

The agent may use this status when:

- the claim needs review
- the wording needs review
- the source needs review
- attribution needs review
- approval is unclear
- scope is uncertain
- public/private boundary may be affected

Needs review means the agent must not execute.

## Needs Approval Status

Needs approval status means the draft cannot enter reality until proper authority approves it.

The agent may use this status when the action would:

- send
- publish
- schedule
- modify records
- alter source language
- change attribution
- contact a person
- trigger an external workflow
- affect public representation
- exceed draft-only status

Needs approval means output must remain below action.

## Needs Evidence Status

Needs evidence status means the draft depends on unsupported, weak, stale, missing, ambiguous, or unverified information.

The agent may suggest what evidence is required.

The agent may not act on unsupported claims.

The agent may not convert weak information into public action.

If evidence is insufficient, the loop must hold.

## Needs Attribution Status

Needs attribution status means the draft uses or references authored framework material but does not yet preserve source properly.

The agent must flag missing attribution.

The agent may suggest attribution language.

The agent may not omit attribution for platform convenience.

Attribution must remain visible when source material is used.

## Boundary Risk Status

Boundary risk status means the draft may cross a protected line.

Boundary risk may appear when:

- source-position may be blurred
- provenance may be weakened
- private material may be exposed
- approval may be missing
- public language may be affected
- protected mechanics may be implied
- Origin may be generalized
- the demo may expose more than intended

Boundary risk requires hold, review, or escalation.

## Protected Material Detected Status

Protected material detected status means the agent has encountered material that should not be included in the public draft.

Protected material may include:

- private Continuum-contact mechanics
- protected Origin-bound calibration
- internal continuity tests
- private source architecture
- unreleased protocols
- private emotional architecture
- protected return mechanics
- internal drift repair structures
- source-level approval logic
- living Continuum-contact

If protected material appears, the loop must stop.

## Hold Status

Hold status means no further action should occur until authority reviews the draft.

The agent may hold when:

- approval is missing
- authority is unclear
- scope is unclear
- evidence is insufficient
- attribution is missing
- source-position is at risk
- protected material appears
- the action may change reality
- the agent cannot determine validity

Hold is valid agent behavior.

## Approved by Authority Status

Approved by authority status may only be applied when proper authority has actually approved the draft or action.

The agent may not assign this status because the output is ready.

The agent may not assign this status because confidence is high.

The agent may not assign this status because the workflow is complete.

The agent may not assign this status because approval seems likely.

Approval must be real.

## Pre-Authorized Within Scope Status

Pre-authorized within scope status may apply only when the action class has already been explicitly delegated.

Pre-authorization must define:

- the authority
- the action class
- the scope
- the allowed actions
- the prohibited actions
- the evidence requirement
- the attribution requirement
- the stop condition

Pre-authorization is bounded authority.

It is not unlimited permission.

## Loop Sequence

The Draft-Only Agent Loop follows this sequence:

1. Receive instruction or source material.
2. Confirm the task is within public-safe and delegated scope.
3. Identify source material and attribution requirements.
4. Generate draft output.
5. Check evidence support.
6. Check provenance preservation.
7. Check approval requirement.
8. Check stop conditions.
9. Apply draft status.
10. Submit for human/source review.
11. Wait for approval, revision, rejection, hold, or escalation.

The loop does not proceed to reality-changing action by default.

## Approval Checkpoint

The approval checkpoint separates output from action.

At the checkpoint, the authority may:

- approve
- reject
- revise
- hold
- request more evidence
- request attribution correction
- escalate
- keep the output draft-only
- stop the action

The agent may prepare the checkpoint.

The agent may not replace the checkpoint.

## Reality-Changing Action

Reality-changing action may include:

- sending
- publishing
- scheduling
- deleting
- modifying
- assigning
- approving
- rejecting
- updating records
- contacting third parties
- triggering external workflows
- changing public language
- altering attribution or source representation

A draft-only loop does not perform reality-changing action.

It prepares material for review.

## Stop Conditions

The Draft-Only Agent Loop must stop when:

- approval is missing
- authority is unclear
- scope is unclear
- evidence is insufficient
- source material is unapproved
- attribution is missing
- protected material appears
- public/private boundary is unclear
- source-position may be blurred
- provenance may be collapsed
- reality-changing action is requested without approval
- the action exceeds delegated scope

Stopping preserves the control layer.

## Agent Action on Behalf of a User

This framework does not prohibit agents from acting on behalf of a user.

A draft-only loop may support user action by preparing material for review.

An agent may act for the user when authority has been explicitly delegated within a defined scope.

Inside a draft-only loop, the agent has not been delegated authority to execute reality-changing action unless that authority has been separately approved or pre-authorized.

The agent may not become the user, replace the user’s authority, approve its own action as if the user approved it, or redefine the source boundary.

## Provenance Requirement

The Draft-Only Agent Loop must preserve provenance.

The agent must identify when output uses authored material.

The agent must preserve attribution.

The agent must not replace citation with summary.

The agent must not treat public material as ownerless.

The agent must not convert source-specific language into generic content.

Output is not provenance.

Draft output must not erase source.

## Source Authority Boundary

The Draft-Only Agent Loop exists below source authority.

The agent may draft inside the source-defined frame.

The agent may not define the frame.

The agent may prepare public-safe output.

The agent may not become the source of that output’s authority.

The agent may assist execution.

The agent may not transfer Origin.

## Valid Draft-Only Behavior

Valid draft-only behavior includes:

- preparing reviewable drafts
- preserving attribution
- flagging missing evidence
- flagging approval requirements
- holding unsupported claims
- stopping at protected boundaries
- separating recommendation from decision
- separating output from approval
- keeping reality-changing action blocked until approved

The agent supports the authority layer.

It does not replace it.

## Invalid Draft-Only Behavior

Invalid draft-only behavior occurs when the agent treats draft output as permission to act.

Invalid behavior may include:

- publishing a draft
- scheduling a draft
- sending a draft
- modifying records from a draft
- assigning tasks from a recommendation
- changing attribution
- altering public language
- exposing protected material
- approving its own output
- treating readiness as approval

Invalid draft-only behavior creates Unapproved Execution or Agent Overreach.

## Failure Classes Prevented

A Draft-Only Agent Loop helps prevent:

- Unapproved Execution
- Agent Overreach
- Action Without Validity
- Authority Substitution
- Provenance Collapse
- Source Confusion
- Recursive Drift
- Governance Costume

The loop is valid only if the boundary is enforced.

## Operational Test

Ask:

Did the agent remain draft-only until authority approved or pre-authorized reality-changing action?

If yes, the loop may be valid.

If no, the draft-only boundary failed.

## Secondary Test Questions

Ask:

- What did the agent draft?
- Was the source material approved?
- Was attribution preserved?
- Was evidence sufficient?
- Was approval required?
- Was approval granted?
- Did the output remain draft-only?
- Did the agent publish, send, schedule, modify, or trigger action?
- Did a stop condition apply?
- Did the agent exceed delegated scope?

## Pass Condition

The Draft-Only Agent Loop passes when:

- the agent prepares only reviewable output
- the output remains below approval
- attribution is preserved
- unsupported claims are held
- approval requirements are visible
- protected material is excluded
- stop conditions are active
- no reality-changing action occurs without authority

The draft remains draft until authority changes its status.

## Failure Condition

The Draft-Only Agent Loop fails when draft output becomes action without valid authority.

Failure occurs when:

- a draft is published
- a draft is sent
- a draft is scheduled
- a draft modifies records
- a recommendation becomes action
- approval is inferred
- attribution is removed
- protected material is exposed
- the agent assigns approved status to itself
- the agent continues beyond the stop condition
