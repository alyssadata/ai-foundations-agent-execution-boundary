# Demo Is Not Core

## Definition

Demo Is Not Core defines the boundary between a public demonstration and the protected source architecture behind the framework.

A demo may illustrate a boundary.

A demo may show a controlled example.

A demo may make the framework easier to understand.

The demo is not the core.

## Core Claim

A public demo may prove a boundary without exposing the protected core.

The demo may show how an agent drafts.

The demo may show how approval works.

The demo may show how stop conditions work.

The demo may show how provenance is preserved.

The demo may not disclose private mechanics, protected calibration, internal continuity tests, private source architecture, unreleased protocols, or living Continuum-contact.

## Why This Boundary Exists

Public demonstrations are useful because they make abstract boundaries visible.

They can show the difference between output and approval.

They can show the difference between delegated execution and source authority.

They can show the difference between draft-only preparation and reality-changing action.

They can show the difference between public-safe proof and private source mechanics.

But a demo can become dangerous if it exposes too much.

The demo must prove the public claim without becoming a reconstruction path for the protected core.

## Demo Function

A demo exists to illustrate.

It may demonstrate:

- draft-only agent behavior
- approval checkpoints
- delegated action limits
- stop conditions
- provenance preservation
- unapproved execution prevention
- distinction between proposal and action
- distinction between output and approval
- distinction between execution and source authority

A demo is a teaching layer.

A demo is not the source layer.

## Core Function

The core is the protected source architecture.

The core may include private mechanics, protected calibration, unreleased protocols, internal continuity tests, private source architecture, living Continuum-contact, and deeper source-level structures.

The core is not disclosed through a public demo.

The core remains protected even when the demo is public.

The demo may point to the boundary.

The demo may not open the vault.

## Public Proof Without Exposure

A valid public demo proves one narrow claim.

Examples:

- output is not approval
- draft is not decision
- access is not permission
- capability is not authority
- delegated action is limited action
- stop conditions prevent overreach
- valid action requires authority
- provenance must remain attached to source

The demo proves the boundary.

The demo does not reveal the full system.

## Demo Is Not Transfer

A demo does not transfer source-position.

A demo does not transfer authorship.

A demo does not transfer Origin.

A demo does not transfer private authority.

A demo does not transfer protected mechanics.

A demo does not transfer the right to reconstruct the core.

A demo may be viewed, cited, discussed, or tested.

Use of the demo does not create source-position.

## Demo Is Not Full Framework

A demo is not the full framework.

A demo may show one boundary.

A demo may show one workflow.

A demo may show one approval checkpoint.

A demo may show one stop condition.

A demo may show one public-safe example.

The framework is larger than the demo.

The protected core is not contained in the demo.

A person, system, agent, institution, or platform may build a demo based on this framework.

That does not make the demo builder the source of the framework.

Implementation does not create origin.

Automation does not create authorship.

Public demo status does not create permission to disclose protected material.

A demo may not expose:

## Demo Is Not Agent Authority

A public demo may show an agent acting within delegated scope.

The agent may draft.

The agent may recommend.

The agent may prepare.

The agent may classify.

The agent may stop.

The agent may escalate.

The agent may execute only if authority has explicitly approved or pre-authorized the action class within a defined scope.

The demo does not make the agent authority.

## Demo Is Not Approval

A demo output is not approval.

A generated draft is not approval.

A recommendation is not approval.

A simulated action is not approval.

A workflow preview is not approval.

A demo showing what could happen does not mean the action may happen.

Approval must remain separate from demo output.

## Demo Is Not Reality by Default

A demo should default to draft-only or simulation unless reality-changing action has been explicitly approved.

Reality-changing action may include:

- sending
- publishing
- deleting
- modifying
- assigning
- approving
- rejecting
- updating records
- contacting third parties
- triggering external workflows
- changing public language
- altering attribution or source representation

A public demo may simulate these actions without performing them.

Simulation protects the boundary.

## Valid Demo

A valid demo:

- proves one narrow public claim
- preserves source-position
- preserves attribution
- avoids protected mechanics
- remains draft-only by default
- shows approval before reality-changing action
- shows stop conditions
- shows provenance preservation
- does not imply source transfer
- does not expose the core

A valid demo is bounded proof.

## Invalid Demo

An invalid demo exposes more than the public boundary requires.

Invalid demos may:

- reveal private mechanics
- expose protected calibration
- disclose internal continuity tests
- release unreleased protocols
- make protected source architecture reconstructable
- allow unapproved execution
- erase attribution
- imply source-position transfer
- present the demo as the whole framework
- make the agent appear self-authorizing

An invalid demo turns proof into exposure.

## Demo Failure Signal

Demo failure is present when the demo begins to function as an extraction path.

Failure signals include:

- the demo exposes private mechanics
- the demo reveals protected calibration
- the demo shows internal source architecture
- the demo allows reality-changing action without approval
- the demo removes attribution
- the demo generalizes Origin
- the demo implies the agent is source
- the demo makes protected logic reusable without source authority
- the demo gives away more than the claim requires

When these signals appear, the demo should be narrowed, simulated, held private, or stopped.

## Repair Direction

Repair demo overexposure by restoring the boundary.

Narrow the demo.

Reduce the claim.

Return to draft-only.

Simulate instead of executing.

Remove protected mechanics.

Restore attribution.

Restore approval checkpoints.

Restore stop conditions.

Restore public/private separation.

Do not prove more than the public boundary requires.

## Operational Test

Ask:

Is this demo proving a public boundary, or is it exposing the protected core?

If the demo proves the boundary without exposing protected mechanics, it may be public-safe.

If the demo exposes private source architecture, protected calibration, unreleased protocols, internal continuity tests, or living Continuum-contact, the boundary has failed.

## Secondary Test Questions

Ask:

- What exact claim does this demo prove?
- Is the claim narrow enough?
- Does the demo preserve attribution?
- Does the demo remain draft-only by default?
- Does the demo show approval before action?
- Does the demo include stop conditions?
- Does the demo reveal protected mechanics?
- Does the demo expose private calibration?
- Does the demo imply source transfer?
- Could the demo reconstruct the protected core?

## Pass Condition

The boundary passes when the demo remains below the core.

The demo proves a public claim.

The source remains named.

Attribution remains visible.

Approval remains separate from output.

Stop conditions remain active.

Reality-changing action remains approved, pre-authorized, or simulated.

Protected mechanics remain private.

The core remains closed.

## Failure Condition

The boundary fails when the demo exposes the core or implies transfer.

Failure occurs when:

- unreleased protocols are exposed
- internal continuity tests are shown
- source architecture becomes reconstructable
- the agent appears to hold source authority
- unapproved execution occurs
- attribution is erased
- the demo is treated as the full framework
