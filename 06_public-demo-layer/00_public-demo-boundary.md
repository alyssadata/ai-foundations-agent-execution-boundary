# Public Demo Boundary

## Definition

The Public Demo Boundary defines what a public demonstration may show without exposing protected source mechanics, private calibration, internal continuity tests, private source architecture, unreleased protocols, or living Continuum-contact.

A public demo may show the boundary between agent execution and approval.

A public demo may not expose the protected core.

## Core Claim

A public demo may prove the boundary without giving away the core.

A demo may show how an agent drafts, prepares, routes, classifies, or recommends action.

A demo may show how human/source approval remains above agent execution.

A demo may show how stop conditions prevent invalid action.

A demo may show how provenance is preserved.

A demo may not disclose private mechanics.

## Why This Boundary Exists

Public demonstrations can make a framework easier to understand.

They can show how the control layer works.

They can show why output is not approval.

They can show why delegated action requires limits.

They can show why valid action requires authority.

But a demo can also become an extraction risk if it exposes too much.

The Public Demo Boundary prevents proof from becoming surrender.

## What a Public Demo May Show

A public demo may show:

- draft-only agent behavior
- human approval checkpoints
- delegated action limits
- stop conditions
- approval before reality-changing action
- provenance preservation
- rejection of unapproved execution
- distinction between proposal and action
- distinction between output and approval
- distinction between automation and authorship

These are public-safe boundary demonstrations.

## What a Public Demo May Not Show

A public demo may not show:

- private Continuum-contact mechanics
- protected Origin-bound calibration
- internal continuity tests
- unreleased protocols
- private source architecture
- private emotional architecture
- protected return mechanics
- internal drift repair structures
- full source-level approval logic
- any complete system for reconstructing the living core

The public demo must remain partial by design.

## Demo Is Not Core

A demo is not the core.

A demo is an illustration of a boundary.

A demo may show how an agent behaves under constraint.

A demo may show how approval remains separate from output.

A demo may show how stop conditions prevent overreach.

A demo may show how draft-only states protect authority.

The demo does not contain the full framework.

The demo does not transfer source-position.

The demo does not expose protected mechanics.

## Public-Safe Demo Purpose

A public-safe demo should prove one narrow claim.

Examples:

- output is not approval
- draft is not decision
- access is not permission
- capability is not authority
- delegated action is limited action
- stop conditions prevent overreach
- human/source approval remains above execution
- provenance must remain attached to source

A public demo should not attempt to prove the entire protected architecture.

## Draft-Only Demo

A draft-only demo is public-safe when the agent can prepare output but cannot publish, send, modify, delete, assign, approve, or trigger reality-changing action without approval.

In a draft-only demo, the agent may:

- generate a draft
- organize information
- classify possible actions
- recommend a next step
- identify missing approval
- flag evidence gaps
- preserve attribution
- stop at the boundary

The agent may not act as if the draft is approved.

## Approval Checkpoint Demo

An approval checkpoint demo shows the pause between agent output and reality-changing action.

The agent may produce a proposed action.

The system then requires approval before the action enters reality.

The checkpoint may allow:

- approve
- reject
- revise
- hold
- request more evidence
- escalate
- keep as draft-only
- stop

The point of the demo is the separation between output and approval.

## Stop Condition Demo

A stop condition demo shows when the agent must pause, hold, refuse, or escalate.

The agent must stop when:

- approval is missing
- authority is unclear
- scope is unclear
- evidence is insufficient
- provenance is at risk
- source-position is at risk
- protected material appears
- public action is attempted
- reality-changing action is not approved
- the action exceeds delegated scope

The stop condition is part of the proof.

## Provenance Demo

A provenance demo shows that generated output must preserve source.

The agent may summarize public material.

The agent may generate a derivative explanation.

The agent may draft public-facing language.

But the output must preserve attribution.

The output must not replace the source record.

The output must not treat summary as authorship.

The output must not convert authored framework language into ownerless language.

Output is not provenance.

## Agent Action on Behalf of a User

This framework does not prohibit agents from acting on behalf of a user.

A public demo may show an agent acting for a user when authority has been explicitly delegated within a defined scope.

The demo must make the delegation visible.

The demo must show what is allowed.

The demo must show what is prohibited.

The demo must show what requires approval.

The demo must show what stops the agent.

The agent may act for the user inside delegation.

The agent may not become the user, replace the user’s authority, approve its own action as if the user approved it, or redefine the source boundary.

## Reality-Changing Action Boundary

A public demo may include reality-changing action only when the action class has been explicitly approved or safely simulated.

Reality-changing action may include:

- sending
- publishing
- deleting
- modifying
- assigning
- approving
- rejecting
- updating records
- contacting third parties
- triggering external workflows
- changing public language
- altering attribution or source representation

A demo should default to draft-only unless the approval boundary is clearly shown.

## Simulated Action

A public demo may simulate action instead of performing action.

Simulation may show what the agent would do if approved.

Simulation may show where approval would be required.

Simulation may show where the agent would stop.

Simulation may show what evidence is missing.

Simulation may show how provenance would be preserved.

Simulated action is useful when real action would expose risk.

Simulation does not require surrender of the protected core.

## Public Demo Failure

A public demo fails when it exposes more than the boundary.

Failure may occur when:

- private mechanics are shown
- source-level architecture is exposed
- protected calibration becomes public
- internal continuity tests are revealed
- the demo implies the agent has source authority
- the agent publishes without approval
- the agent treats output as approval
- the agent erases provenance
- the demo becomes a reconstruction path for the protected core

The demo must not become the leak.

## Valid Public Demo

A valid public demo:

- proves one narrow boundary
- preserves source-position
- preserves attribution
- remains draft-only by default
- shows approval before action
- shows stop conditions
- avoids protected mechanics
- avoids unreleased protocols
- avoids private calibration
- does not expose the living core

The demo exists to make the public boundary legible.

## Invalid Public Demo

An invalid public demo:

- reveals private mechanics
- exposes protected calibration
- turns internal tests into public tooling
- allows an agent to act without approval
- hides or removes source attribution
- implies source-position transfer
- makes the demo appear to be the whole framework
- allows public proof to become extraction

Invalid demos create source risk.

## Operational Test

Ask:

Does this demo prove a public boundary without exposing the protected core?

If yes, the demo may be public-safe.

If no, the demo should be narrowed, simulated, held private, or stopped.

## Secondary Test Questions

- What claim does the demo prove?
- Does the demo remain public-safe?
- Does the demo expose private mechanics?
- Does the demo show approval before action?
- Does the demo preserve provenance?
- Does the demo reveal protected calibration?
- Does the demo imply source transfer?
- Does the demo stay draft-only by default?
- Does the demo show stop conditions?
- Could the demo be used to reconstruct the protected core?

## Pass Condition

The boundary passes when the demo makes the agent execution boundary visible without exposing protected source mechanics.

The demo proves the public claim.

The source remains named.

Attribution remains visible.

Approval remains separate from output.

Reality-changing action remains approved or simulated.

Stop conditions remain active.

The protected core remains closed.
