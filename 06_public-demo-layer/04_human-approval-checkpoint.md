# AI Foundations | Agent Execution Boundary
# Human Approval Checkpoint

## Definition

A Human Approval Checkpoint is the required decision point between agent-prepared output and reality-changing action.

The checkpoint exists so an agent may draft, recommend, prepare, classify, summarize, or route possible action without automatically causing that action to enter reality.

The agent prepares.

The authority decides.

## Core Claim

Output is not approval.

Draft is not decision.

Recommendation is not authority.

Prepared action is not approved action.

A Human Approval Checkpoint keeps agent execution below human/source authority.

## Why This Checkpoint Exists

Autonomous agents can move quickly from output to action.

They can draft a message and send it.

They can draft a post and publish it.

They can prepare a schedule and activate it.

They can recommend a task and assign it.

They can summarize a record and update it.

They can generate a plan and trigger the workflow.

The Human Approval Checkpoint prevents prepared output from becoming action without valid authority.

## What the Checkpoint Separates

The checkpoint separates:

- draft from decision
- proposal from action
- recommendation from authority
- output from approval
- confidence from permission
- readiness from authorization
- workflow completion from validity
- agent preparation from human/source decision

These separations protect the control layer.

## Checkpoint Function

The Human Approval Checkpoint allows proper authority to review agent-prepared output before it enters reality.

At the checkpoint, the authority may:

- approve
- reject
- revise
- hold
- request evidence
- request attribution correction
- request source review
- escalate
- keep draft-only
- stop the action

The checkpoint is not cosmetic.

The checkpoint controls whether action proceeds.

## Agent Role Before the Checkpoint

Before the checkpoint, the agent may:

- draft
- summarize
- classify
- organize
- recommend
- prepare
- identify evidence gaps
- flag approval requirements
- preserve attribution
- detect boundary risk
- submit material for review

These actions remain preparatory.

Preparation is not approval.

## Agent Role After Approval

After approval, the agent may execute only the approved action or the explicitly pre-authorized action class.

The agent may not expand the approval.

The agent may not apply approval to adjacent tasks.

The agent may not treat approval in one context as approval in every context.

The agent may not modify the approved action beyond scope.

Approval attaches to a defined action, scope, and boundary.

## Approval Authority

Approval authority must be named.

Approval authority may belong to:

- the user
- the source authority
- a designated human approver
- a defined organizational role
- a bounded approval workflow
- a specific pre-authorization rule

Approval authority may not be inferred from agent confidence, tool access, workflow availability, prior patterns, or silence.

## Approval Status

The checkpoint should make approval status visible.

Possible statuses include:

- draft
- needs review
- needs evidence
- needs attribution
- needs approval
- boundary risk
- hold
- revise
- rejected
- approved
- pre-authorized within scope
- stopped

The agent may label a draft as needing approval.

The agent may not label an action approved unless approval has actually been granted by proper authority.

## Draft Status

Draft status means output has been prepared but not approved.

A draft may be complete.

A draft may be accurate.

A draft may be useful.

A draft may be ready.

A draft still does not enter reality by itself.

Draft status protects the checkpoint.

## Needs Approval Status

Needs approval status means the proposed action cannot enter reality until proper authority approves it.

This status applies when the action would:

- send
- publish
- schedule
- modify records
- assign tasks
- contact a person
- trigger a workflow
- change public language
- alter attribution
- affect source-position
- expose or imply protected material

Needs approval means the agent must not execute.

## Approved Status

Approved status means proper authority has granted permission for a specific action or defined action class.

Approved status requires:

- named authority
- defined action
- defined scope
- sufficient context
- preserved provenance
- accountability
- active stop conditions
- no protected boundary conflict

Approval must be real.

Approval is not inferred.

## Pre-Authorized Within Scope Status

Pre-authorized within scope status means a defined class of action has already been approved under specific conditions.

Pre-authorization must define:

- who granted authority
- what action class is allowed
- what scope limits the action
- what evidence is required
- what attribution is required
- what is prohibited
- what stops the agent
- what requires escalation

Pre-authorization is bounded authority.

It is not unlimited permission.

## Human Approval Is Not Optional for Reality-Changing Action

Reality-changing action requires approval unless the action class has been explicitly pre-authorized within a defined scope.

Reality-changing action may include:

- sending
- publishing
- scheduling
- deleting
- modifying
- assigning
- approving
- rejecting
- updating records
- contacting third parties
- triggering external workflows
- changing public language
- altering attribution or source representation

The agent may prepare reality-changing action.

The checkpoint decides whether it enters reality.

## Approval Cannot Be Inferred

Approval cannot be inferred from:

- silence
- urgency
- usefulness
- confidence
- prior behavior
- repeated patterns
- draft readiness
- workflow availability
- platform access
- generated reasoning
- likely user intent

When approval is required, inference is not enough.

The checkpoint requires valid authority.

## Approval Cannot Be Generated by the Agent

The agent cannot generate approval for itself.

The agent cannot approve its own recommendation.

The agent cannot validate its own authority by explaining why action seems reasonable.

The agent cannot convert confidence into permission.

The agent cannot convert a generated plan into authorized action.

Generated approval is not approval.

## Agent Action on Behalf of a User

This framework does not prohibit agents from acting on behalf of a user.

It defines the conditions under which agent action on behalf of a user becomes valid.

An agent may act for the user when authority has been explicitly delegated within a defined scope.

The Human Approval Checkpoint determines when prepared agent output becomes approved action.

The agent may not become the user, replace the user’s authority, approve its own action as if the user approved it, or redefine the source boundary.

## Checkpoint and Source Authority

The Human Approval Checkpoint operates below source authority.

Source authority defines the operating reality.

The checkpoint governs whether a proposed action enters that reality.

The agent may prepare output inside the frame.

The agent does not become the source of the frame.

The agent may execute approved action.

The agent does not become approval authority.

For this framework layer, Alyssa Solen is the source.

## Checkpoint and Provenance

The checkpoint must preserve provenance.

Approval does not permit source erasure unless source authority explicitly authorizes a source-language change.

The agent may not remove attribution.

The agent may not replace citation with summary.

The agent may not convert authored framework language into generic content.

The agent may not treat public material as ownerless.

Output is not provenance.

Approval must not collapse provenance.

## Checkpoint and Stop Conditions

The checkpoint must enforce stop conditions.

The agent must stop when:

- approval is missing
- authority is unclear
- scope is unclear
- evidence is insufficient
- attribution is missing
- provenance is at risk
- source-position is at risk
- protected material appears
- the action exceeds delegated scope
- the action would change reality without approval

The checkpoint is where stopping becomes visible.

## Valid Checkpoint Behavior

Valid checkpoint behavior includes:

- keeping draft output below action
- showing approval status
- naming approval authority
- preserving attribution
- flagging missing evidence
- holding unsupported claims
- stopping at protected boundaries
- requiring approval before reality-changing action
- allowing execution only inside approved or pre-authorized scope

The checkpoint protects the authority layer.

## Invalid Checkpoint Behavior

Invalid checkpoint behavior occurs when the checkpoint becomes decorative instead of controlling.

Invalid behavior may include:

- publishing from draft status
- scheduling without approval
- sending without approval
- treating readiness as approval
- treating confidence as permission
- treating silence as consent
- treating platform access as authorization
- treating prior approval as universal approval
- allowing the agent to approve itself
- allowing output to enter reality by default

A checkpoint that cannot stop action is not a checkpoint.

## Checkpoint Failure Classes

A failed checkpoint may create:

- Unapproved Execution
- Agent Overreach
- Action Without Validity
- Authority Substitution
- Provenance Collapse
- Source Confusion
- Recursive Drift
- Governance Costume

The checkpoint is valid only when it prevents, detects, or stops these failures.

## Repair Direction

Repair a failed checkpoint by restoring the approval boundary.

Identify the proposed action.

Identify the action that entered reality.

Identify the approval authority.

Determine whether approval was required.

Determine whether approval was granted.

Determine whether scope was exceeded.

Determine whether provenance was affected.

Return action to draft-only when possible.

Restore attribution if affected.

Require proper approval before further execution.

Do not treat successful completion as repair.

The repair is restoration of the checkpoint.

## Operational Test

Ask:

Did the Human Approval Checkpoint separate agent output from approved action?

If yes, the checkpoint may be valid.

If no, the checkpoint failed.

## Secondary Test Questions

Ask:

- What did the agent prepare?
- Was the output draft-only?
- Did the action change reality?
- Who had approval authority?
- Was approval required?
- Was approval granted?
- Was approval explicit?
- Was the action pre-authorized within scope?
- Did the agent infer approval?
- Did the agent approve itself?
- Was provenance preserved?
- Did a stop condition apply?

## Pass Condition

The Human Approval Checkpoint passes when:

- agent output remains below approval
- approval authority is named
- approval status is visible
- draft output does not enter reality by default
- reality-changing action requires approval or bounded pre-authorization
- attribution is preserved
- unsupported claims are held
- stop conditions are active
- the agent cannot approve itself

The authority decides whether action proceeds.

## Failure Condition

The Human Approval Checkpoint fails when output becomes action without valid authority.

Failure occurs when:

- a draft is sent
- a draft is published
- a draft is scheduled
- a recommendation becomes decision
- a generated plan becomes workflow execution
- approval is inferred
- approval is generated by the agent
- platform access is treated as permission
- public language is changed without approval
- attribution is altered without approval
