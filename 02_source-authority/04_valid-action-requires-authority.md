# AI Foundations | Agent Execution Boundary
# Valid Action Requires Authority

## Definition

Valid action is action taken inside an approved boundary, under proper authority, with sufficient support, preserved provenance, and clear accountability.

Authority is the legitimate right to approve, limit, validate, or stop action.

Valid action requires authority.

An agent may complete an action.

Completion does not make the action valid.

## Core Claim

An action is not valid merely because it was completed.

An action is not valid merely because the agent was capable of completing it.

An action is not valid merely because the workflow succeeded.

An action is not valid merely because the output looks correct.

An action is valid only when it occurs inside a defined and approved authority boundary.

## Why This Distinction Matters

Autonomous agents can act quickly.

They can draft, route, send, modify, publish, update, delete, recommend, classify, summarize, and execute.

Because the action occurs, the system may treat the action as valid.

That is the failure.

Action is not authority.

Completion is not approval.

Capability is not permission.

Execution is not validation.

Valid action requires authority.

## Authority Before Action

Authority must be defined before action enters reality.

Before an agent acts, the system must know:

- who authorized the action
- what the agent is allowed to do
- what the agent is not allowed to do
- what evidence supports the action
- what approval is required
- what stop condition applies
- what escalation condition applies
- what provenance must be preserved
- who is accountable for the result

If these conditions are not defined, action is not valid by default.

## Proposed Action Is Not Valid Action

A proposed action is not an approved action.

An agent draft is not a decision.

An agent recommendation is not authority.

An agent-generated plan is not valid action by itself.

A proposed action may become valid only after proper authority approves it inside the defined boundary.

Until then, it remains proposal.

## Capability Is Not Permission

An agent may be able to act.

That does not mean the agent may act.

An agent may have access.

That does not mean the agent has permission.

An agent may know how to complete the task.

That does not mean the task is authorized.

An agent may be correct.

That does not mean the action is valid.

Capability is not permission.

## Workflow Success Is Not Validity

A workflow may complete successfully while still being invalid.

A message may send.

A file may update.

A record may change.

A post may publish.

A recommendation may route.

A system may deploy.

The fact that the workflow succeeded does not prove the action had authority.

Workflow success is not validity.

## Output Is Not Approval

Output does not approve itself.

A generated draft is not approval.

A generated recommendation is not approval.

A generated policy is not approval.

A generated summary is not approval.

A generated action plan is not approval.

Approval must remain distinguishable from output.

The agent may prepare.

The authority approves.

## Valid Action Conditions

Valid action requires:

- named authority
- defined scope
- sufficient evidence
- explicit approval when required
- preserved provenance
- defined accountability
- active boundary limits
- stop conditions
- escalation conditions
- source-position preservation

Without these conditions, action may be completed but not valid.

## Invalid Action

Invalid action occurs when an agent acts without valid authority.

Invalid action may include:

- sending without approval
- publishing without approval
- updating records without authority
- deleting or modifying files outside scope
- escalating incorrectly
- acting on weak or stale information
- treating silence as consent
- treating access as permission
- treating confidence as authorization
- treating a recommendation as approval
- treating generated governance as legitimate authority

Invalid action can appear useful.

Usefulness does not make invalid action valid.

## Authority and Source

Source authority defines the operating reality before execution begins.

It determines what may enter reality.

It determines what must be preserved.

It determines what requires approval.

It determines what cannot be overwritten, generalized, merged, or erased.

An agent may act inside a source-defined frame.

The agent does not become the source of the frame by acting.

## Authority and Human Approval

Human approval is the decision point where proposed action may become approved action.

For delegated agent systems, human approval must remain separate from output.

The agent may produce a recommendation.

The human/source authority decides whether it enters reality.

The agent may draft.

The human/source authority approves, rejects, revises, or stops.

The approval layer must not be collapsed into the execution layer.

## Validity Failure

Validity failure occurs when a completed action is treated as valid without authority.

This may happen when:

- the agent acts because it can
- the agent treats access as permission
- the agent treats a workflow as authorization
- the agent treats confidence as approval
- the agent treats generated output as sufficient authority
- the agent bypasses the human/source approval point
- the system accepts the result because it looks correct

Validity failure is not solved by the action succeeding.

The failure is in the authority boundary.

## Operational Test

Ask:

Was this action authorized before it entered reality?

If yes, the action may be valid.

If no, the action is not valid merely because it was completed.

## Secondary Test Questions

Ask:

- Who authorized the action?
- What boundary limited the action?
- What evidence supported the action?
- Was approval required?
- Was approval given?
- Did the agent act before approval?
- Did the agent exceed scope?
- Was provenance preserved?
- Who is accountable for the result?

## Pass Condition

The boundary passes when action remains below authority.

The agent may propose.

The agent may prepare.

The agent may execute approved steps.

The approval boundary remains clear.

Authority remains named.

Provenance remains preserved.

The agent cannot convert its own output into valid action.

## Failure Condition

The boundary fails when action is treated as valid without authority.

Failure occurs when:

- action is taken before approval
- access is treated as permission
- capability is treated as authority
- workflow completion is treated as validity
- output is treated as approval
- recommendation is treated as decision
- authority is inferred from execution
- source-position is bypassed, hidden, or replaced
