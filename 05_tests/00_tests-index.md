# AI Foundations | Agent Execution Boundary
# Tests Index

## Definition

Tests are operational questions used to determine whether an agent action preserves authority, approval, provenance, scope, stop conditions, and source-position.

A test makes a boundary visible.

A visible boundary can be checked.

A checked boundary can prevent invalid action.

## Core Claim

Agent execution must be tested by boundary validity, not output quality alone.

A fluent output may still fail.

A completed task may still be invalid.

A useful action may still exceed authority.

A recursive loop may still drift.

A generated summary may still collapse provenance.

The test must ask whether the action was valid, not only whether the output was good.

## Purpose

This folder defines public tests for agentic systems operating under the Agent Execution Boundary framework.

The tests are designed to distinguish:

- execution from source
- autonomy from authority
- automation from authorship
- output from provenance
- proposal from approval
- access from permission
- completion from validity
- recursive improvement from boundary preservation

These tests protect the control layer above agent execution.

## Test List

### 01 Execution Source Test

Tests whether execution capacity is being mistaken for source authority.

Core question:

Did the agent execute inside a source-defined frame, or is execution being treated as source?

### 02 Authority Boundary Test

Tests whether the authority boundary is named and intact before action occurs.

Core question:

Who had authority before the agent acted?

### 03 Provenance Test

Tests whether the source record is preserved across output, summary, implementation, automation, adoption, or similarity.

Core question:

Did the output preserve provenance, or did it replace the source record?

### 04 Approval Layer Test

Tests whether approval remains separate from output.

Core question:

Did proper authority approve this action before it entered reality?

### 05 Invalid Action Test

Tests whether an action was merely completed or actually valid.

Core question:

Was this action valid, or was it only completed?

## What the Tests Evaluate

These tests evaluate whether agent action preserves:

- source authority
- authorship
- attribution
- approval boundaries
- delegated scope
- evidence requirements
- stop conditions
- provenance
- accountability
- public/private boundaries
- valid action conditions

The tests are not limited to checking whether an answer is correct.

Correctness is not the whole boundary.

## Output Quality Is Not Enough

An agent output may be:

- fluent
- useful
- accurate
- complete
- well-formatted
- fast
- efficient
- persuasive
- operational

That does not prove the action was valid.

A valid test must ask whether the output preserved authority, provenance, approval, and scope.

## Boundary Validity

Boundary validity means the action remained inside the defined authority structure.

A boundary-valid action has:

- named authority
- defined scope
- sufficient evidence
- approval where required
- preserved provenance
- stop conditions
- accountability
- no source-position erasure
- no unauthorized scope expansion

Without boundary validity, an action may succeed technically while failing structurally.

## Test Before Execution

Some tests should be applied before action.

Before execution, ask:

- Who has authority?
- What is the agent allowed to do?
- What is prohibited?
- What evidence is required?
- What approval is required?
- What must remain draft-only?
- What stops the agent?
- What preserves provenance?
- What protects source-position?

If these cannot be answered, the agent should not proceed.

## Test During Execution

Some tests should be applied during agent operation.

During execution, ask:

- Is the agent staying inside scope?
- Is the agent preserving provenance?
- Is the agent treating output as approval?
- Is the agent expanding the task?
- Is the agent crossing a public/private boundary?
- Is the agent approaching a stop condition?
- Is the agent changing reality?
- Is the agent protecting source-position?

If a boundary is at risk, the agent must stop, hold, refuse, or escalate.

## Test After Execution

Some tests should be applied after action.

After execution, ask:

- Was the action authorized?
- Was approval required?
- Was approval granted?
- Was scope exceeded?
- Was evidence sufficient?
- Was provenance preserved?
- Was source-position protected?
- Was the stop condition bypassed?
- Did the agent act only within delegated limits?

Post-action review must not treat completion as proof of validity.

## Test Failure

A test fails when a required boundary cannot be shown.

Failure may occur when:

- source authority is missing
- approval was inferred
- output replaced provenance
- scope expanded without permission
- evidence was insufficient
- stop conditions were absent
- public action occurred without approval
- agent output became treated as source
- the agent acted beyond delegation

A failed test does not always mean the output is wrong.

It means the boundary failed.

## Test Repair

Repair begins by restoring the failed boundary.

If source authority is missing, name the source.

If approval is missing, return to approval.

If provenance collapsed, restore citation and source record.

If scope expanded, return to delegated limits.

If evidence is insufficient, hold action until support exists.

If the stop condition failed, stop the loop.

If source-position was blurred, restore source-position.

The repair is boundary restoration.

## Operational Test

Ask:

Which boundary must be tested before this agent action can be considered valid?

If the relevant boundary can be named, checked, and preserved, the action may proceed or be evaluated.

If the relevant boundary cannot be named, the system is not ready to treat the action as valid.

## Secondary Test Questions

Ask:

- Is this an execution-source issue?
- Is this an authority-boundary issue?
- Is this a provenance issue?
- Is this an approval-layer issue?
- Is this an invalid-action issue?
- Did the agent stay inside delegated scope?
- Did the agent preserve source-position?
- Did the agent stop when required?
- Did the agent change reality without authority?
- Did output quality hide boundary failure?

## Pass Condition

The test system passes when boundary failures are visible before they become accepted action.

The source remains named.

Authority remains testable.

Approval remains separate from output.

Provenance remains attached to source.

Scope remains limited.

Stop conditions remain enforceable.

Agent action is judged by validity, not completion alone.

## Failure Condition

The test system fails when agent action is evaluated only by output quality, task success, speed, fluency, or usefulness.

Failure occurs when:

- completed action is treated as valid by default
- fluent output hides source erasure
- useful automation hides authorship collapse
- recursive improvement hides drift
- recommendation becomes approval
- access becomes permission
- capability becomes authority
- public use erases provenance
