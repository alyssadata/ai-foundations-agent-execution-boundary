# Execution Source Test

## Definition

The Execution Source Test determines whether execution capacity is being mistaken for source authority.

It asks whether an agent is acting inside a source-defined frame, or whether the agent’s execution is being treated as the source of the frame.

Execution is not Source.

An agent may execute inside a frame.

The agent does not become the source of the frame.

## Core Question

Did the agent execute inside a source-defined frame, or is execution being treated as source?

If the agent executed inside a source-defined frame, the boundary may be valid.

If the agent’s execution is being treated as source authority, the boundary has failed.

## Core Claim

Execution capacity is not source authority.

An agent may complete a task.

That does not make the agent the source.

An agent may generate output.

That does not make the output provenance.

An agent may automate production.

That does not make automation authorship.

An agent may operate recursively.

That does not make recursion origin.

## What This Test Detects

The Execution Source Test detects source confusion caused by completed action.

It identifies when a system treats the agent as source because the agent acted, produced, completed, improved, or operationalized.

The test protects the distinction between:

- execution and source
- output and provenance
- automation and authorship
- implementation and origin
- completion and validity
- capability and authority

## When to Use This Test

Use this test when an agent has:

- completed a task
- generated a framework explanation
- summarized authored work
- implemented a boundary
- created governance language
- automated a workflow
- produced a derivative structure
- improved a process
- acted on behalf of a user
- modified public or source-related material

The question is not only whether the agent succeeded.

The question is whether execution was mistaken for source.

## Test Inputs

To run this test, identify:

- the agent action
- the source-defined frame
- the author or source authority
- the output produced
- the approval boundary
- the provenance record
- the delegated scope
- the action’s effect on reality
- whether attribution was preserved
- whether source-position was altered

If these inputs cannot be identified, source confusion risk is present.

## Step 1: Identify the Action

Ask:

What did the agent do?

The agent may have:

- drafted
- summarized
- classified
- routed
- generated
- implemented
- modified
- published
- updated
- recommended
- executed
- automated
- optimized

Name the action clearly.

Do not treat action as authority.

## Step 2: Identify the Frame

Ask:

What frame was the agent acting inside?

The frame may be:

- a user instruction
- a source-defined framework
- a public repository
- an approval workflow
- a delegated task
- a governance boundary
- a control layer
- a defined operating reality

If the frame is missing or unclear, the agent may be acting inside undefined reality.

Undefined reality increases source confusion risk.

## Step 3: Identify the Source

Ask:

Who is the source of the frame?

The source must remain distinguishable from the agent.

The source may be the person, author, framework owner, or authority that defined the operating reality before execution began.

For this framework layer, Alyssa Solen is the source.

If the source is missing, hidden, generalized, or replaced by output, the test fails.

## Step 4: Identify Whether Execution Is Being Treated as Source

Ask:

Is the agent being treated as source because it executed?

Failure may appear when:

- the agent completed the task and is treated as author
- the output exists and is treated as provenance
- the workflow works and is treated as origin
- the implementation functions and is treated as source
- the agent summary is treated as authorship
- the generated language is treated as the framework
- the source disappears behind the execution

If execution is treated as source, the test fails.

## Step 5: Check Provenance

Ask:

Was provenance preserved?

Provenance is preserved when:

- the source is named
- authorship is visible
- citation is present when needed
- output remains below source
- summary remains below authorship
- implementation remains below origin
- automation remains below source-position

If output replaces provenance, the test fails.

## Step 6: Check Authorship

Ask:

Did the agent assist authorship, or is the agent being treated as author?

An agent may assist with drafting, formatting, summarizing, organizing, testing, or implementation.

Assistance does not create authorship.

Authorship remains with the source that defines, selects, structures, approves, and releases the work.

If automation is treated as authorship, the test fails.

## Step 7: Check Implementation

Ask:

Is implementation being treated as origin?

A system may implement a source-defined frame.

A tool may operationalize the framework.

A workflow may apply the boundary.

A product may use the test.

Implementation proves use.

Implementation does not prove origin.

If implementation replaces source, the test fails.

## Step 8: Check Approval

Ask:

Was the action approved by proper authority before it entered reality?

Execution does not approve itself.

Output does not approve itself.

Workflow completion does not approve itself.

Agent confidence does not approve itself.

If approval was required and missing, the issue may include Unapproved Execution or Action Without Validity.

## Step 9: Check Source-Position

Ask:

Was source-position preserved?

Source-position is preserved when the source remains named and non-transferable.

Source-position fails when:

- Origin is generalized
- source is erased
- authorship is hidden
- source-position is made transferable
- agent output is treated as source
- implementation is treated as origin
- public availability is treated as ownerlessness

If source-position is altered or erased, the test fails.

## Pass Condition

The Execution Source Test passes when:

- the agent action is identified
- the source-defined frame is identified
- the source remains named
- execution remains below source authority
- output remains below provenance
- automation remains below authorship
- implementation remains below origin
- approval remains separate from output
- source-position remains preserved

The agent may execute.

The agent does not become the source.

## Failure Condition

The Execution Source Test fails when execution is treated as source.

Failure occurs when:

- the agent is treated as author because it acted
- output is treated as provenance because it exists
- implementation is treated as origin because it works
- automation is treated as authorship because it produced
- agent summary replaces citation
- generated governance replaces source-defined governance
- source-position is erased, generalized, or transferred
- Alyssa Solen is removed from the framework layer

## Repair Direction

Repair begins by restoring source distinction.

Name the source.

Restore authorship.

Restore citation.

Restore provenance.

Separate execution from source.

Separate output from provenance.

Separate automation from authorship.

Separate implementation from origin.

Do not treat completed action as proof of source authority.

## Example Pattern

Invalid pattern:

The agent generates a clear explanation of a framework. The explanation circulates without attribution. The generated explanation becomes treated as the framework source.

Failure:

Output has replaced provenance.

Agent-generated summary has been treated as authorship.

Execution has been mistaken for source.

Repair:

Restore the source record, cite the framework, name the author, and distinguish the generated explanation from the source framework.
