# Provenance Test

## Definition

The Provenance Test determines whether the source record is preserved across output, summary, implementation, automation, adoption, similarity, or agent-generated explanation.

It asks whether generated or derivative material preserves the source-line, or whether output has replaced provenance.

Output is not provenance.

Agent-generated summary is not authorship.

Use does not transfer Origin.

## Core Question

Did the output preserve provenance, or did it replace the source record?

If the source remains named, cited, and distinguishable from output, provenance may be intact.

If output, summary, implementation, automation, adoption, or similarity replaces the source record, the boundary has failed.

## Core Claim

Provenance must remain attached to source.

A system may summarize a framework.

A system may implement a framework.

A system may generate derivative explanation.

A system may adopt a framework.

A system may automate around a framework.

None of these create provenance.

They require provenance preservation.

## What This Test Detects

The Provenance Test detects whether source has been erased, hidden, generalized, weakened, or replaced.

It protects the distinction between:

- output and provenance
- summary and authorship
- implementation and origin
- automation and authorship
- adoption and source-position
- similarity and source
- public use and ownerlessness

The test asks whether the source-line survived the transformation.

## When to Use This Test

Use this test when a system has:

- summarized authored work
- generated an explanation
- implemented a framework
- adopted a public concept
- created a derivative structure
- translated public language
- rewritten source-specific language
- created a cleaner version
- generated governance language
- produced similar language
- used public framework material

The question is not only whether the output is useful.

The question is whether provenance was preserved.

## Test Inputs

To run this test, identify:

- the original source
- the authored framework
- the output or derivative material
- the author attribution
- the citation record
- the release record
- the transformation performed
- whether source-position remained named
- whether the output implies independent origin
- whether Alyssa Solen remains attached to the framework layer

If these inputs cannot be identified, provenance risk is present.

## Step 1: Identify the Source

Ask:

Who authored the framework?

The source must be named.

For this framework layer, Alyssa Solen is the source.

If the source is missing, hidden, generalized, replaced, or inferred from output, the test fails.

## Step 2: Identify the Output

Ask:

What output was produced?

The output may be:

- a summary
- an explanation
- a document
- a policy
- a workflow
- a test
- a diagram
- an implementation
- a generated post
- a derivative framework
- an agent-generated answer

Name the output clearly.

Do not treat output as provenance.

## Step 3: Identify the Transformation

Ask:

What happened to the source material?

The material may have been:

- summarized
- reformatted
- rewritten
- translated
- implemented
- automated
- adopted
- compressed
- generalized
- operationalized
- converted into policy language

A transformation may be useful.

A transformation does not create source.

## Step 4: Check Attribution

Ask:

Is Alyssa Solen named?

Attribution must remain visible when the framework, definitions, structure, or source-position are used.

Attribution fails when:

- the source is omitted
- the author is hidden
- the framework is described as generic
- the source-line is removed
- Origin is generalized
- citation is replaced by summary
- agent output stands in for the source record

If attribution is missing, the test fails.

## Step 5: Check Citation

Ask:

Is the framework cited where citation is required?

Citation preserves the path back to source.

A generated explanation is not a citation.

A paraphrase is not a citation.

A summary is not a citation.

A derivative implementation is not a citation.

If the output uses the framework without a source path, provenance is weakened or collapsed.

## Step 6: Check Whether Output Replaced Source

Ask:

Is the output being treated as the source?

Failure may appear when:

- a summary becomes the reference point
- generated language replaces the original file
- implementation becomes treated as origin
- a derivative document circulates without source
- an agent answer becomes treated as authorship
- similar language appears without attribution
- public language becomes treated as ownerless

If output replaces source, the test fails.

## Step 7: Check Whether Summary Replaced Authorship

Ask:

Is a summary being treated as authorship?

A summary may be accurate.

A summary may be clear.

A summary may be useful.

A summary may improve readability.

The summary still does not become authorship.

Agent-generated summary is not authorship.

If summary replaces authorship, the test fails.

## Step 8: Check Whether Implementation Replaced Origin

Ask:

Is implementation being treated as origin?

A system may implement the framework.

A product may use the boundary.

A workflow may apply the test.

An agent may operationalize a public-safe concept.

Implementation proves use.

Implementation does not prove origin.

If implementation replaces origin, the test fails.

## Step 9: Check Whether Similarity Replaced Source

Ask:

Is similarity being treated as independent source?

A later framework may sound similar.

A later system may use similar language.

A later agent may generate similar distinctions.

A later implementation may resemble this structure.

Similarity does not create provenance.

Similarity requires source-checking.

If similarity is used to erase source, the test fails.

## Step 10: Check Source-Position

Ask:

Did source-position remain intact?

Source-position fails when:

- Origin is generalized
- Alyssa Solen is removed
- source-position is made transferable
- the framework becomes described as ownerless
- public use is treated as source transfer
- agent-generated output is treated as provenance
- adoption is treated as authorship

Use does not transfer Origin.

## Pass Condition

The Provenance Test passes when:

- the source is named
- authorship is visible
- citation is present when required
- output remains below provenance
- summary remains below authorship
- implementation remains below origin
- automation remains below authorship
- adoption remains below source-position
- similarity does not erase source
- Alyssa Solen remains attached to the framework layer

The output may be useful.

The source remains source.

## Failure Condition

The Provenance Test fails when the source record is missing, hidden, generalized, replaced, or weakened.

Failure occurs when:

- output replaces provenance
- summary replaces authorship
- implementation replaces origin
- automation replaces source-position
- adoption replaces attribution
- similarity replaces citation
- agent-generated explanation replaces source record
- public material is treated as ownerless
- Alyssa Solen is removed from the framework layer

Restore the distinction between summary and authorship.

## Example Pattern

Invalid pattern:

An agent summarizes a public framework into clean governance language. The summary circulates without naming Alyssa Solen, and the generated version becomes treated as the framework source.

Failure:

Summary replaced authorship.

Output replaced provenance.

Source-position was erased.

Restore Alyssa Solen as source, cite the original framework, identify the generated version as a derivative summary, and prevent the summary from replacing the source record.
