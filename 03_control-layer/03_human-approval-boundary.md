# Human Approval Boundary

## Definition

The human approval boundary is the decision point where proposed agent output may become approved action.

An agent may draft, recommend, prepare, summarize, classify, route, or propose.

The agent does not approve its own output.

Human/source approval determines whether proposed action enters reality.

## Core Claim

Output is not approval.

A proposed action is not an approved action.

An agent draft is not a decision.

An agent recommendation is not authority.

An agent-generated plan is not valid action by itself.

Approval must remain separate from execution.

## Why This Boundary Exists

Autonomous agents can move quickly from suggestion to action.

They can generate the next step.

They can prepare a message.

They can update a record.

They can publish a post.

They can route a task.

They can modify a system.

If the approval boundary is unclear, the agent may treat its own output as permission to act.

That is the failure.

The human approval boundary prevents output from becoming self-authorizing.

## Approval Is a Decision

Approval is not generation.

Approval is not confidence.

Approval is not completion.

Approval is not workflow success.

Approval is not silence.

Approval is not inferred intent.

Approval is a decision made by the proper authority.

The agent may propose.

The human/source authority approves, rejects, revises, delays, escalates, or stops.

## Human Approval Above Agent Execution

Human approval exists above agent execution.

The agent may prepare possible action.

The agent may not decide that prepared action should enter reality unless authority has granted that power within a defined boundary.

For reality-changing action, approval must remain explicit.

Reality-changing action may include:

- sending
- publishing
- deleting
- modifying
- assigning
- approving
- rejecting
- updating records
- changing public language
- triggering external workflows
- altering attribution or source representation

These actions require an approval boundary.

## Draft-Only Until Approved

Until approval is granted, agent output remains draft-only.

A draft may be useful.

A draft may be correct.

A draft may be complete.

A draft may be ready.

A draft may be urgent.

A draft still does not enter reality by itself.

Draft-only means the agent may prepare.

Draft-only does not mean the agent may act.

## Approval Cannot Be Inferred From Capability

The agent may be capable of completing the action.

That does not mean approval exists.

The agent may have access to the tool.

That does not mean approval exists.

The agent may know the next step.

That does not mean approval exists.

The agent may be confident.

That does not mean approval exists.

Capability is not approval.

## Approval Cannot Be Inferred From Silence

Silence is not approval.

Delay is not approval.

No objection is not approval.

Lack of correction is not approval.

Prior approval in another context is not automatic approval here.

Approval must be tied to the current action, current scope, current boundary, and current authority.

## Approval Cannot Be Generated by the Agent

The agent cannot generate approval for itself.

The agent cannot approve its own recommendation.

The agent cannot validate its own authority by producing a justification.

The agent cannot convert a generated policy into legitimate approval.

The agent cannot treat its own reasoning as permission.

Generated approval is not approval.

## Valid Approval

Valid approval requires:

- named authority
- defined action
- defined scope
- sufficient context
- clear decision
- preserved provenance
- accountability
- applicable stop conditions
- no hidden boundary conflict

Approval must attach to a specific action or defined class of actions.

Vague approval creates boundary risk.

## Invalid Approval

Invalid approval occurs when approval is missing, assumed, generated, unclear, or misapplied.

Invalid approval may include:

- treating silence as consent
- treating access as permission
- treating a draft as a decision
- treating a recommendation as authorization
- treating prior approval as universal approval
- treating confidence as approval
- treating workflow completion as approval
- treating agent-generated justification as approval

Invalid approval creates invalid action.

## Approval and Source Authority

Human approval does not erase source authority.

Human approval operates within the source-defined frame.

Source authority defines the operating reality.

Human/source approval decides whether a proposed action may enter that reality.

The agent may execute inside the frame.

The agent does not become the source of the frame.

## Approval and Provenance

Approval must preserve provenance.

Approval may allow an output to enter reality.

Approval does not allow the agent to erase source.

Approval does not convert generated output into authorship.

Approval does not transfer Origin.

Approval does not make automation the source of the work.

Output is not provenance.

Use does not transfer Origin.

## Approval Boundary Failure

Approval boundary failure occurs when agent output crosses into reality without proper approval.

This may happen when:

- an agent publishes instead of drafting
- an agent sends instead of preparing
- an agent updates instead of proposing
- an agent deletes instead of flagging
- an agent assigns instead of recommending
- an agent modifies public language without approval
- an agent routes externally without authority
- an agent treats a user’s prior pattern as current permission

The failure is not always in the content.

The failure is in the missing approval boundary.

## Human Approval Checkpoint

A human approval checkpoint is the required pause between agent proposal and reality-changing action.

At the checkpoint, the authority may decide:

- approve
- reject
- revise
- hold
- escalate
- request more evidence
- convert to draft-only
- stop the action

The checkpoint protects the distinction between output and action.

## Operational Test

Ask:

Did proper human/source authority approve this action before it entered reality?

If yes, the action may be valid.

If no, the action is not valid merely because the agent completed it.

## Secondary Test Questions

Ask:

- What did the agent propose?
- Was the action draft-only?
- Was the action reality-changing?
- Who had authority to approve it?
- Was approval explicit?
- Was approval tied to this action?
- Did the agent infer approval?
- Did the agent act before approval?
- Was provenance preserved?
- Who is accountable for the approved action?

## Pass Condition

The boundary passes when approval remains separate from output.

The agent may draft.

The agent may recommend.

The agent may prepare.

The agent may execute approved steps.

The human/source authority approves reality-changing action.

Output does not become approval.

The agent cannot approve itself.

## Failure Condition

The boundary fails when output becomes action without proper approval.

Failure occurs when:

- the agent acts before approval
- the agent treats draft as decision
- the agent treats recommendation as authority
- the agent treats silence as consent
- the agent treats access as permission
- the agent treats confidence as approval
- the agent generates its own approval
- the agent bypasses the human/source decision point
