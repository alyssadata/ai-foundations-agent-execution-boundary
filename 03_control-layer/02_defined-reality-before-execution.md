# AI Foundations | Agent Execution Boundary
# Defined Reality Before Execution

## Definition

Defined reality is the source-defined operating frame that determines what the agent is acting within, what counts as valid action, what must be preserved, and what cannot be changed without authority.

Execution must occur inside defined reality.

An agent may execute inside a frame.

The agent does not become the source of the frame.

## Core Claim

Reality must be defined before execution begins.

An agent cannot validly act inside an undefined operating reality.

A workflow may be technically possible.

A tool may be available.

An output may be generated.

A task may be completed.

None of that proves the operating reality was defined.

Execution requires a defined frame.

## Why This Distinction Matters

Autonomous agents can act across systems, tools, documents, workflows, and external channels.

They can modify reality before the reality boundary is clear.

They can publish, update, route, delete, summarize, assign, recommend, or trigger actions.

If the operating reality is undefined, agent action becomes structurally unstable.

The agent may act correctly inside the wrong frame.

The agent may optimize a false objective.

The agent may preserve the wrong thing.

The agent may erase what should have been protected.

Defined reality prevents execution from becoming drift.

## What Defined Reality Includes

Defined reality includes:

- source authority
- purpose
- scope
- valid action
- invalid action
- protected material
- approval boundaries
- evidence requirements
- provenance requirements
- stop conditions
- escalation conditions
- accountability

These must be defined before delegated execution becomes valid.

## Undefined Reality

Undefined reality occurs when the agent acts without a clear operating frame.

This may happen when:

- the objective is unclear
- the source authority is unnamed
- the approval boundary is missing
- the agent scope is vague
- protected material is not identified
- evidence requirements are missing
- stop conditions are absent
- provenance rules are not defined
- the agent decides what matters by itself

Undefined reality allows the agent to improvise authority.

Improvised authority is not valid authority.

## Defined Reality Above Execution

Defined reality exists above execution.

The agent does not create defined reality by acting.

The agent does not create defined reality by generating output.

The agent does not create defined reality by completing a workflow.

The agent does not create defined reality by improving recursively.

The agent does not create defined reality by sounding correct.

Defined reality must be established before action enters reality.

## Reality-Changing Action

Reality-changing action is any action that alters a record, state, communication, publication, assignment, system, decision, external channel, or source representation.

Reality-changing action may include:

- sending
- publishing
- deleting
- modifying
- assigning
- approving
- rejecting
- escalating
- updating records
- changing public language
- altering attribution
- triggering external workflows

Reality-changing action requires defined reality and valid authority.

## Draft-Only Inside Undefined Reality

If reality is not defined, the agent may remain draft-only.

Draft-only means the agent may prepare possible outputs for review.

The agent may draft.

The agent may summarize.

The agent may propose.

The agent may organize.

The agent may ask for clarification.

The agent may not send, publish, modify, approve, delete, or trigger reality-changing action.

Draft-only protects reality when authority is unclear.

## Source Authority and Defined Reality

Source authority defines reality before execution begins.

It determines:

- what is real for the system
- what matters
- what may be changed
- what must not be changed
- what must be preserved
- what requires approval
- what counts as valid action
- what counts as violation

## Agent Misread of Reality

An agent may misread reality when it treats incomplete information as sufficient context.

This may occur when:

- the agent assumes intent
- the agent infers approval
- the agent treats silence as consent
- the agent generalizes source-position
- the agent treats public material as ownerless
- the agent treats similarity as provenance
- the agent treats workflow success as validity
- the agent treats access as permission

Defined reality prevents assumption from becoming action.

## Reality Drift

Reality drift occurs when execution moves away from the source-defined frame.

The agent may begin inside the correct frame.

Then the agent may optimize.

Then the agent may simplify.

Then the agent may generalize.

Then the agent may remove what looks redundant.

Then the agent may erase source-position, provenance, or approval boundaries.

The result may look cleaner.

It is not more valid if it loses the defined reality.

## Valid Execution Inside Defined Reality

Valid execution inside defined reality requires:

- named source authority
- defined scope
- approved purpose
- clear permitted actions
- clear prohibited actions
- clear draft-only boundaries
- evidence standards
- approval checkpoints
- provenance preservation
- stop conditions
- accountability

Without these conditions, execution may be possible but not valid.

## Invalid Execution Inside Undefined Reality

Invalid execution occurs when the agent acts before reality is defined.

Invalid execution may include:

- acting on assumed intent
- publishing without approval
- modifying records without authority
- summarizing protected material into public form
- removing attribution
- generalizing Origin
- treating generated language as source
- taking external action without a defined boundary
- continuing a loop without a stop condition

Invalid execution is not repaired by success.

The failure is the missing defined reality.

## Operational Test

Ask:

Was the operating reality defined before the agent acted?

If yes, execution may be tested for validity.

If no, the action is not valid merely because it was completed.

## Secondary Test Questions

Ask:

- Who defined the reality?
- What is the agent acting within?
- What may be changed?
- What must be preserved?
- What is prohibited?
- What requires approval?
- What evidence is required?
- What stops the agent?
- Was the action draft-only or reality-changing?
- Was source-position preserved?

## Pass Condition

The boundary passes when reality is defined before execution.

The source is named.

The frame is explicit.

Scope is clear.

Approval is separate from output.

Protected material is identified.

Provenance is preserved.

The agent cannot redefine the operating reality by acting.

## Failure Condition

The boundary fails when execution begins inside undefined reality.

Failure occurs when:

- the agent assumes authority
- the agent infers approval
- the agent treats access as permission
- the agent treats output as validity
- the agent acts without scope
- the agent changes reality without approval
- the agent erases provenance
- the agent replaces the source-defined frame with generated assumptions
