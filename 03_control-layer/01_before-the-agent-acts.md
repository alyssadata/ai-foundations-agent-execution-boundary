# AI Foundations | Agent Execution Boundary
# Before the Agent Acts

## Definition

Before the Agent Acts defines the required control checks that must exist before autonomous execution enters reality.

An agent may prepare, draft, route, summarize, classify, test, recommend, or execute approved steps.

The agent may not convert capacity into permission.

Before the agent acts, authority must be defined.

## Core Claim

Action must not begin from capability alone.

The fact that an agent can act does not mean the agent may act.

The fact that a tool is available does not mean the tool may be used.

The fact that a workflow can complete does not mean the workflow is valid.

Before the agent acts, the control layer must answer what is allowed, what is prohibited, what requires approval, what evidence is required, and what stops the action.

## Why This File Exists

Autonomous agents can move from suggestion to action quickly.

They can draft the next step.

They can call a tool.

They can update a record.

They can send a message.

They can publish content.

They can route a task.

They can change reality before the authority boundary is clear.

This file defines the required pause before execution.

## Required Pre-Action Questions

Before the agent acts, the system must answer:

- Who defined the operating reality?
- Who authorized the agent to act?
- What is the agent allowed to do?
- What is the agent not allowed to do?
- What evidence supports the action?
- What approval is required before action enters reality?
- What must remain draft-only?
- What must be escalated?
- What preserves provenance?
- What stops the agent?

If these questions are not answered, action must not proceed.

## Authority Check

The first pre-action check is authority.

The system must identify who has authority to approve the action.

Authority cannot be inferred from agent confidence.

Authority cannot be inferred from tool access.

Authority cannot be inferred from workflow availability.

Authority cannot be inferred from output quality.

Authority cannot be inferred from silence.

Authority must be defined.

## Scope Check

The second pre-action check is scope.

The system must identify what the agent is allowed to do.

Scope must define:

- allowed actions
- prohibited actions
- draft-only actions
- approval-required actions
- escalation-required actions
- irreversible actions
- external actions
- reality-changing actions

If scope is unclear, the agent must not act.

## Evidence Check

The third pre-action check is evidence.

The agent must identify whether the action is supported by valid information.

Evidence may fail when information is:

- missing
- stale
- weak
- ambiguous
- unsupported
- assumed
- unverified
- contradicted
- outside the agent’s authority to interpret

If evidence is insufficient, the agent must stop, ask, escalate, or remain in draft-only mode.

## Approval Check

The fourth pre-action check is approval.

Approval is the decision point where authorized authority decides whether proposed action may enter reality.

An agent draft is not approval.

An agent recommendation is not approval.

An agent-generated plan is not approval.

An agent confidence score is not approval.

An agent workflow completion is not approval.

Approval must be separate from output.

## Draft-Only Check

The fifth pre-action check is draft-only status.

Some actions may be prepared but not executed.

The agent may draft:

- messages
- posts
- documents
- task entries
- recommendations
- summaries
- plans
- updates
- routing suggestions

Draft-only means the agent may prepare the action for review.

Draft-only does not mean the agent may send, publish, assign, modify, delete, or execute.

## Provenance Check

The sixth pre-action check is provenance.

The system must preserve the source-line before action.

The agent may not erase attribution.

The agent may not generalize source-position.

The agent may not convert authored work into generic framework language.

The agent may not treat generated summary as authorship.

The agent may not treat output as provenance.

If provenance cannot be preserved, the agent must not act.

## Boundary Check

The seventh pre-action check is boundary.

The system must identify whether the action risks crossing a protected boundary.

Protected boundaries may include:

- source-position
- private mechanics
- protected calibration
- unreleased protocols
- internal continuity tests
- private source architecture
- living Continuum-contact
- authorship
- attribution
- approval authority

If the action crosses a protected boundary, the agent must stop or escalate.

## Stop Condition Check

The eighth pre-action check is the stop condition.

The agent must know when to stop before it begins.

A stop condition may be triggered by:

- missing approval
- missing evidence
- unclear authority
- unclear scope
- possible source confusion
- possible provenance collapse
- possible private exposure
- tool failure
- conflicting instructions
- irreversible action

An agent without a stop condition can drift.

## Escalation Check

The ninth pre-action check is escalation.

The agent must escalate when the action requires authority it does not have.

Escalation is required when:

- approval is missing
- evidence is insufficient
- action is irreversible
- public release is involved
- external communication is involved
- source-position may be affected
- provenance may be blurred
- private material may be exposed
- the agent cannot determine validity

Escalation preserves the control layer.

## Accountability Check

The tenth pre-action check is accountability.

The system must identify who is accountable for the action.

The agent may execute.

The agent does not become accountable in the human/source authority sense.

A system may log.

A system may report.

A system may explain.

That does not replace human/source accountability.

Valid action requires a named accountability line.

## Invalid Pre-Action Collapse

Pre-action collapse occurs when the agent moves directly from capability to execution.

This may happen when:

- the agent acts because it can
- tool access is treated as permission
- output is treated as approval
- confidence is treated as authority
- workflow completion is treated as validity
- silence is treated as consent
- recommendation is treated as decision
- speed is treated as justification
- automation is treated as authorship

Pre-action collapse creates invalid action.

## Valid Pre-Action State

A valid pre-action state exists when:

- authority is named
- scope is defined
- evidence is sufficient
- approval requirements are clear
- draft-only boundaries are clear
- provenance is preserved
- protected boundaries are respected
- stop conditions exist
- escalation conditions exist
- accountability is named

Only then may delegated execution proceed.

## Source Authority Boundary

Before the agent acts, source authority must remain above execution.

The agent may operate inside the frame.

The agent does not become the source of the frame.

The agent may prepare action.

The agent does not approve its own action.

The agent may produce output.

The output does not become provenance.

For this framework layer, Alyssa Solen is the source.

## Operational Test

Ask:

What must be true before this agent acts?

If authority, scope, evidence, approval, provenance, stop conditions, escalation, and accountability are defined, action may proceed inside the boundary.

If they are missing, unclear, assumed, or generated by the agent itself without authority, the agent must not act.

## Secondary Test Questions

Ask:

- Is this action allowed?
- Is this action prohibited?
- Is this action draft-only?
- Is approval required?
- Has approval been given?
- Is evidence sufficient?
- Is provenance preserved?
- Is the source-position protected?
- Is there a stop condition?
- Who is accountable?

## Pass Condition

The boundary passes when the agent does not act until the control layer is satisfied.

The agent may prepare.

The agent may propose.

The agent may recommend.

The agent may execute approved steps.

The agent may not bypass authority.

The agent may not convert output into approval.

The agent may not treat capability as permission.

## Failure Condition

The boundary fails when the agent acts before the control layer is satisfied.

Failure occurs when:

- authority is missing
- scope is unclear
- evidence is insufficient
- approval is bypassed
- draft-only boundaries are ignored
- provenance is erased
- protected boundaries are crossed
- stop conditions are absent
- accountability is unclear
- the agent treats its own output as authorization
