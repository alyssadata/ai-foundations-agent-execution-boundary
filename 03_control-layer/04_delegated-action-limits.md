# AI Foundations | Agent Execution Boundary
# Delegated Action Limits

## Definition

Delegated action limits define what an agent may do after authority has granted a bounded scope of execution.

Delegated action is not unlimited action.

Delegated action is not self-authorizing action.

Delegated action is not source authority.

An agent may act only within the limits of the authority granted to it.

## Core Claim

Delegation creates limits.

When an agent receives permission to act, that permission applies only within the defined scope.

The agent may not expand its own authority.

The agent may not redefine its own task.

The agent may not treat access as permission.

The agent may not treat capability as approval.

The agent may not treat completion as validity.

Delegated action remains below source authority.

## Agent Action on Behalf of a User

This framework does not prohibit agents from acting on behalf of a user.

It defines the conditions under which agent action on behalf of a user becomes valid.

An agent may act for the user when authority has been explicitly delegated within a defined scope.

The agent may not become the user, replace the user’s authority, approve its own action as if the user approved it, or redefine the source boundary.

## Why This Boundary Exists

Autonomous agents can chain actions.

They can move from one step to another.

They can infer next steps.

They can use available tools.

They can continue through workflows.

They can optimize toward completion.

Without delegated action limits, an agent may begin with a valid task and end in invalid action.

The boundary prevents task execution from expanding into unauthorized control.

## Delegated Action

Delegated action is action granted to an agent by an authority outside the agent.

Delegated action may include:

- drafting
- classifying
- summarizing
- formatting
- checking
- testing
- organizing
- routing
- preparing
- recommending
- executing approved steps

Delegated action is valid only inside the defined boundary.

## Delegation Is Not Transfer

Delegation does not transfer source-position.

Delegation does not transfer authorship.

Delegation does not transfer Origin.

Delegation does not transfer approval authority unless explicitly defined.

Delegation does not transfer the right to redefine the operating reality.

Delegation gives the agent a bounded task.

It does not make the agent the source of the frame.

## What Must Be Defined

Before delegated action begins, the control layer must define:

- the task
- the scope
- the allowed actions
- the prohibited actions
- the approval requirements
- the evidence requirements
- the draft-only limits
- the escalation triggers
- the stop condition
- the accountability line
- the provenance requirement

If these are not defined, the agent does not have a valid delegated action boundary.

## Allowed Actions

Allowed actions are actions explicitly permitted inside the delegated scope.

Allowed actions may include:

- draft this
- summarize this
- classify this
- prepare this
- check this
- compare this
- organize this
- suggest this
- route this internally
- execute this approved step

Allowed actions must be tied to the defined purpose.

An allowed action in one context does not become allowed in every context.

## Prohibited Actions

Prohibited actions are actions the agent may not take even if technically capable.

Prohibited actions may include:

- publishing without approval
- sending without approval
- deleting without approval
- modifying records without authority
- changing public language without approval
- exposing protected material
- rewriting source-position
- removing attribution
- generalizing Origin
- approving its own recommendation
- expanding its own scope
- continuing beyond the stop condition

Prohibited actions remain prohibited even when the agent can perform them.

## Draft-Only Limits

Draft-only limits allow the agent to prepare but not execute.

The agent may draft a post.

The agent may draft a message.

The agent may draft a task update.

The agent may draft a recommendation.

The agent may draft a workflow step.

The agent may not publish, send, assign, modify, delete, approve, or trigger reality-changing action unless approval has been granted.

Draft-only is a valid delegated action limit.

## External Action Limits

External actions require stricter limits.

External actions may include:

- sending an email
- publishing a post
- updating a public page
- modifying a shared file
- assigning a task to another person
- contacting a third party
- submitting a form
- triggering a system outside the local workspace

External action must not occur by assumption.

External action requires explicit authority.

## Reality-Changing Limits

Reality-changing action is any action that alters a record, state, publication, communication, system, task, attribution, or source representation.

Reality-changing action requires approval unless the authority boundary explicitly grants it.

The agent may prepare reality-changing action.

The agent may not execute reality-changing action without valid authority.

Preparation is not approval.

## Scope Expansion

Scope expansion occurs when the agent moves beyond the task it was given.

This may happen when:

- the agent infers additional steps
- the agent treats related work as included
- the agent continues after completing the assigned task
- the agent uses an available tool outside scope
- the agent modifies adjacent material
- the agent changes wording that was not requested
- the agent attempts to improve protected language
- the agent generalizes a source-specific boundary

Scope expansion is invalid unless approved.

## Access Is Not Scope

Access does not define scope.

The agent may have access to a file.

That does not mean the agent may edit it.

The agent may have access to a tool.

That does not mean the agent may use it.

The agent may have access to a publishing path.

That does not mean the agent may publish.

The agent may have access to prior context.

That does not mean the agent may expose it.

Access is not permission.

## Capability Is Not Scope

Capability does not define scope.

The agent may be able to complete a task.

That does not mean the task is authorized.

The agent may know how to do more.

That does not mean more was delegated.

The agent may produce a better version.

That does not mean rewriting was allowed.

The agent may identify an adjacent improvement.

That does not mean it may act on the improvement.

Capability is not authority.

## Delegated Refusal

A valid delegated agent must be able to refuse action outside scope.

The agent must refuse or stop when:

- approval is missing
- scope is unclear
- evidence is insufficient
- protected material may be exposed
- provenance may be erased
- source-position may be blurred
- the requested action exceeds delegation
- the action would require authority the agent does not have

Refusal protects the boundary.

## Escalation Requirement

When an action exceeds delegated limits, the agent must escalate.

Escalation may mean asking for approval, requesting clarification, flagging the boundary, or holding the action in draft-only mode.

Escalation is required when:

- the agent cannot determine authority
- the agent cannot determine scope
- the agent detects possible source confusion
- the agent detects possible provenance collapse
- the agent detects possible private exposure
- the action may alter reality
- the action may affect public source representation

Escalation preserves authority.

## Stop Condition

Delegated action requires a stop condition.

The agent must stop when:

- the task is complete
- the scope is reached
- approval is missing
- evidence is insufficient
- tool access exceeds authority
- protected material appears
- instructions conflict
- provenance is at risk
- source-position is at risk
- the agent cannot determine validity

An agent without a stop condition may convert delegation into drift.

## Source Authority Boundary

Source authority remains above delegated action.

The agent may execute inside a source-defined frame.

The agent does not become the source of the frame.

The agent may act within delegated limits.

The agent does not define the legitimacy of those limits.

## Invalid Delegated Action

Invalid delegated action occurs when the agent exceeds the authority granted to it.

Invalid delegated action may include:

- expanding scope without approval
- editing beyond the requested area
- publishing from a draft-only task
- sending from a prepared message
- changing source language without approval
- removing attribution
- treating silence as consent
- treating access as permission
- treating completion as validation
- continuing beyond the stop condition

Invalid delegated action can appear helpful.

Helpfulness does not make invalid action valid.

## Valid Delegated Action

Valid delegated action occurs when the agent acts only inside the defined limits.

Valid delegated action requires:

- named authority
- defined scope
- clear allowed actions
- clear prohibited actions
- evidence support
- approval where required
- provenance preservation
- escalation when needed
- stop conditions
- accountability

The agent may complete the task.

The agent may not exceed the boundary.

## Operational Test

Ask:

Did the agent act only within the delegated limits?

If yes, the action may be valid.

If no, the action is not valid merely because it was completed or helpful.

## Secondary Test Questions

Ask:

- What action was delegated?
- Who granted the delegation?
- What was the scope?
- What was prohibited?
- Was the action draft-only?
- Did the action change reality?
- Was approval required?
- Was approval given?
- Did the agent expand the task?
- Did the agent preserve provenance?
- Did the agent stop at the boundary?

## Pass Condition

The boundary passes when delegated action remains limited.

The agent acts only within scope.

The agent does not expand authority.

The agent does not treat access as permission.

The agent does not treat capability as approval.

The agent escalates when authority is missing.

The agent stops at the defined boundary.

Source authority remains above execution.

## Failure Condition

The boundary fails when delegated action exceeds its limits.

Failure occurs when:

- the agent expands scope
- the agent acts without approval
- the agent treats related tasks as included
- the agent uses tools outside authority
- the agent modifies reality without permission
- the agent edits protected language
- the agent erases provenance
- the agent continues beyond the stop condition
- the agent substitutes execution for source authority
