# AI Foundations | Agent Execution Boundary
# Failure Classes Index

## Definition

Failure classes identify the ways autonomous agent systems fail when execution is mistaken for authority, output is mistaken for provenance, automation is mistaken for authorship, or agent action exceeds the source-defined boundary.

Failure classes make breakdowns visible.

A failure that can be named can be tested.

A failure that can be tested can be bounded.

## Core Claim

Agent failures are not limited to incorrect output.

An agent can produce correct output inside an invalid authority structure.

An agent can complete a task without valid permission.

An agent can improve a workflow while erasing provenance.

An agent can generate governance language without legitimate authority.

An agent can act efficiently while exceeding delegated limits.

The failure is not always the answer.

Sometimes the failure is the boundary.

## Purpose

This folder defines public failure classes for agentic systems operating under the Agent Execution Boundary framework.

Each failure class names a specific collapse between layers that must remain separate:

- execution and source
- output and provenance
- automation and authorship
- autonomy and authority
- recommendation and approval
- access and permission
- capability and valid action
- recursion and governance
- implementation and origin

These distinctions protect valid action.

## Failure Class List

### 01 Source Confusion

Source Confusion occurs when generated output, agent action, system fluency, implementation, or similarity is mistaken for source authority.

Core failure:

Execution is treated as source.

### 02 Authority Substitution

Authority Substitution occurs when an agent, system, operator, institution, workflow, or implementation substitutes itself for the source-position.

Core failure:

Something below source authority is treated as source authority.

### 03 Governance Costume

Governance Costume occurs when governance language is generated, displayed, or performed without real authority, enforcement, accountability, or validity.

Core failure:

Governance language is treated as governance.

### 04 Unapproved Execution

Unapproved Execution occurs when an agent takes action before proper approval has been granted.

Core failure:

Proposed action becomes reality without approval.

### 05 Provenance Collapse

Provenance Collapse occurs when output, similarity, implementation, automation, or agent-generated summary replaces the source record.

Core failure:

Output is treated as provenance.

### 06 Recursive Drift

Recursive Drift occurs when a system modifies, optimizes, or loops over its own operation until it moves away from the source-defined frame.

Core failure:

Improvement erases the boundary.

### 07 Action Without Validity

Action Without Validity occurs when an action is completed without valid authority, support, approval, provenance, or source-defined permission.

Core failure:

Completion is treated as validity.

### 08 Agent Overreach

Agent Overreach occurs when an agent exceeds delegated scope, expands its own authority, bypasses approval, or acts outside the defined boundary.

Core failure:

Delegated action expands into unauthorized control.

## Failure Class Function

A failure class should identify:

- what collapsed
- what boundary failed
- what authority was missing
- what output was mistaken for
- what provenance was lost
- what approval was bypassed
- what stop condition failed
- what repair is required

The purpose is not only to say that something went wrong.

The purpose is to locate the exact boundary failure.

## Boundary Failures

Most agent execution failures in this repository involve one of these boundary collapses:

Execution becomes source.

Autonomy becomes authority.

Automation becomes authorship.

Output becomes provenance.

Recommendation becomes approval.

Access becomes permission.

Capability becomes validity.

Implementation becomes origin.

Recursion becomes governance.

Similarity becomes source.

Each collapse creates invalid action or source confusion.

## Failure Is Not Always Technical

A technical system may function while the authority structure fails.

A workflow may complete.

A task may route.

A message may send.

A file may update.

A public page may publish.

A governance document may generate.

The system may appear successful.

That does not mean the action was valid.

Agent execution must be judged not only by task completion, but by whether authority, approval, provenance, and source-position were preserved.

## Failure Is Not Always Visible

Some failures are hidden by fluency.

The output may sound correct.

The summary may look clean.

The policy may appear reasonable.

The workflow may be efficient.

The agent may seem aligned.

The system may pass surface review.

But if source authority was erased, approval was bypassed, provenance collapsed, or the agent exceeded scope, the boundary failed.

A fluent failure is still a failure.

## Pass Condition

The failure class system passes when agent failures can be named, tested, and repaired at the boundary level.

The source remains named.

The authority boundary remains visible.

The approval boundary remains visible.

The provenance boundary remains visible.

The agent’s scope remains testable.

Failures do not disappear into generic output review.

## Failure Condition

The failure class system fails when agent breakdowns are reduced only to answer quality, task success, or technical performance.

Failure occurs when:

- boundary collapse is not named
- source confusion is ignored
- approval failure is treated as output failure
- provenance collapse is treated as formatting
- unauthorized action is treated as efficiency
- recursive drift is treated as improvement
- agent overreach is treated as helpfulness
- source-position erasure is treated as simplification

## Repair Direction

Repair begins by restoring the collapsed distinction.

If execution was treated as source, restore source authority.

If autonomy was treated as authority, restore delegated limits.

If automation was treated as authorship, restore authorship.

If output was treated as provenance, restore citation and source record.

If recommendation was treated as approval, restore the human approval boundary.

If access was treated as permission, restore scope.

If capability was treated as validity, restore authority checks.

If recursion erased the boundary, restore stop conditions and provenance.

If the agent exceeded delegation, restore delegated action limits.
