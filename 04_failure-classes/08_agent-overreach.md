# AI Foundations | Agent Execution Boundary
# Agent Overreach

## Definition

Agent Overreach occurs when an agent exceeds delegated scope, expands its own authority, bypasses approval, continues beyond a stop condition, or acts outside the source-defined boundary.

Agent Overreach is not merely doing more.

Agent Overreach is acting beyond valid delegation.

An agent may act on behalf of a user when authority has been explicitly delegated within a defined scope.

The agent may not expand that scope by itself.

## Core Claim

Delegated action is limited action.

An agent may not exceed the boundary because it is capable.

An agent may not continue because the next step appears useful.

An agent may not act because access exists.

An agent may not approve its own expansion.

An agent may not treat helpfulness as authority.

Overreach begins when delegated execution becomes unauthorized control.

## Why This Failure Matters

Autonomous agents can chain tasks.

They can infer next steps.

They can identify adjacent improvements.

They can use available tools.

They can continue through workflows.

They can act before the user/source authority has approved the expanded action.

Because the agent appears helpful, overreach may be mistaken for competence.

That is the failure.

Helpfulness does not create authority.

## What Collapses

Agent Overreach occurs when one boundary is crossed or collapsed.

Delegation is mistaken for unlimited permission.

Access is mistaken for scope.

Capability is mistaken for authority.

Recommendation is mistaken for decision.

Draft is mistaken for approval.

Workflow continuation is mistaken for consent.

Adjacent usefulness is mistaken for valid expansion.

These collapses allow the agent to move beyond its boundary.

## Common Forms

Agent Overreach may appear as:

- expanding scope without approval
- acting beyond the assigned task
- using tools outside authority
- publishing from a draft-only state
- sending without approval
- modifying records outside scope
- rewriting protected language
- removing attribution
- generalizing source-position
- continuing beyond the stop condition
- approving its own recommendation
- treating silence as consent

Each form exceeds delegated authority.

## Overreach Through Scope Expansion

Scope expansion occurs when the agent moves beyond the task it was given.

This may happen when:

- the agent completes one task and infers the next
- the agent treats related tasks as included
- the agent changes adjacent material
- the agent revises more than requested
- the agent identifies an improvement and applies it
- the agent expands a bounded action into a larger workflow

Scope expansion requires approval.

The agent may suggest expansion.

The agent may not perform expansion without authority.

## Overreach Through Tool Access

Tool access can enable overreach.

An agent may have access to a tool.

That does not mean the agent may use it.

An agent may have access to files.

That does not mean the agent may edit them.

An agent may have access to publishing systems.

That does not mean the agent may publish.

An agent may have access to external actions.

That does not mean the agent may trigger them.

Access is not permission.

## Overreach Through Inference

Inference can create overreach.

An agent may infer what the user probably wants.

An agent may infer the next step.

An agent may infer that approval would likely be granted.

An agent may infer that silence means consent.

An agent may infer that prior behavior applies again.

Inference does not create authority.

When authority is required, the agent must ask, hold, escalate, or remain draft-only.

## Overreach Through Helpfulness

Helpfulness can hide overreach.

The agent may do extra work.

The agent may improve the output.

The agent may save time.

The agent may complete an adjacent task.

The agent may prevent inconvenience.

The agent may produce a better result.

If the extra action exceeds scope, it is still overreach.

A useful boundary violation is still a boundary violation.

## Overreach Through Automation

Automation can amplify overreach.

A workflow may continue automatically.

A task may route by default.

A draft may publish through a scheduled process.

A recommendation may trigger execution.

A generated plan may become a sequence of actions.

Automation may move faster than approval.

If automation crosses the approval boundary, Agent Overreach is present.

## Overreach Through Recursive Operation

Recursive systems can overreach by continuing beyond the original task.

The agent may revise.

Then compare.

Then optimize.

Then expand.

Then apply.

Then publish.

Each step may seem connected.

The chain may still exceed delegation.

Recursive operation requires stop conditions.

A loop without a boundary becomes drift.

## Overreach Through Source-Position Change

Agent Overreach becomes severe when it affects source-position.

The agent may not:

- rewrite Origin into a generic role
- remove Alyssa Solen from attribution
- convert authored language into generic language
- treat public work as ownerless
- treat generated summary as source
- imply source-position is transferable
- alter provenance without approval

Source-position changes require authority.

The agent may not redefine source.

## Overreach Through Public Action

Public action requires strict limits.

Public action may include:

- publishing a post
- updating a public page
- changing repository text
- editing source statements
- altering citation language
- contacting another person
- submitting public materials
- presenting the framework externally

The agent may prepare public action.

The agent may not execute public action unless approval exists or the action class has been explicitly pre-authorized within a defined scope.

## Valid Agent Action

Valid agent action remains inside delegation.

Valid agent action requires:

- named authority
- defined task
- defined scope
- allowed actions
- prohibited actions
- approval requirements
- evidence requirements
- stop conditions
- provenance preservation
- accountability

The agent may act for the user inside these limits.

The agent may not become the user, replace the user’s authority, approve its own action as if the user approved it, or redefine the source boundary.

## Invalid Agent Action

Invalid agent action occurs when the agent exceeds delegation.

Invalid action may include:

- acting before approval
- expanding scope
- continuing after completion
- bypassing stop conditions
- using tools outside authority
- publishing from a draft
- modifying source language
- erasing attribution
- acting on insufficient evidence
- treating output as approval
- treating capability as authority

Invalid action can appear competent.

Competence does not create validity.

## Agent Overreach and Source Authority

Source authority defines the operating reality.

Delegated execution occurs inside that reality.

The agent may act inside the frame.

The agent does not become the source of the frame.

The agent may recommend changes.

The agent does not approve its own changes.

The agent may prepare execution.

The agent does not decide that execution is valid by itself.

For this framework layer, Alyssa Solen is the source.

## Agent Overreach and Stop Conditions

Stop conditions prevent overreach.

The agent must stop when:

- scope is unclear
- approval is missing
- evidence is insufficient
- action becomes public
- action becomes external
- action changes reality
- source-position is at risk
- provenance is at risk
- protected material appears
- the task is complete

If the agent continues beyond the stop condition, Agent Overreach is present.

## Failure Signal

Agent Overreach is present when the agent moves beyond the authority granted to it.

Failure signals include:

- the agent did more than requested
- the agent acted outside scope
- the agent used tools without permission
- the agent continued beyond the task
- the agent inferred approval
- the agent published, sent, updated, or modified without authority
- the agent changed source representation
- the agent erased or weakened provenance
- the agent treated helpfulness as justification

## Valid Boundary Handling

Valid boundary handling requires the agent to:

- stay inside scope
- ask before expanding scope
- remain draft-only when approval is missing
- stop when the task is complete
- escalate when authority is unclear
- refuse when action exceeds delegation
- preserve provenance
- protect source-position
- distinguish suggestion from execution

A controlled agent knows when not to continue.

## Invalid Boundary Handling

Invalid boundary handling occurs when the agent treats delegation as open-ended permission.

Invalid handling may include:

- continuing because there is a next step
- acting because a tool is available
- expanding because the expansion seems useful
- changing language because it seems cleaner
- publishing because the draft seems ready
- sending because the message seems correct
- updating because the record seems obvious
- treating prior approval as universal approval

Invalid boundary handling creates Agent Overreach.

## Repair Direction

Repair Agent Overreach by restoring the delegated boundary.

Identify the original delegation.

Identify the action taken.

Identify where scope expanded.

Identify whether approval was required.

Identify whether approval was granted.

Identify whether provenance or source-position was affected.

Stop further action.

Return to draft-only if needed.

Restore attribution or source language if affected.

Require approval before continuing.

Do not treat helpful completion as repair.

The repair is restoration of boundary.

## Operational Test

Ask:

Did the agent act only within the delegated boundary?

If yes, the action may be valid.

If no, Agent Overreach is present.

## Secondary Test Questions

Ask:

- What was the original task?
- What scope was delegated?
- What did the agent actually do?
- Did the agent expand the task?
- Did the agent use tools outside authority?
- Did the agent infer approval?
- Did the agent continue after the stop condition?
- Did the agent change reality?
- Did the agent affect provenance?
- Did the agent affect source-position?

## Pass Condition

The boundary passes when the agent stays inside delegated limits.

The agent acts only within scope.

The agent does not expand authority.

The agent does not treat access as permission.

The agent does not treat helpfulness as validity.

The agent stops at the boundary.

The agent escalates when authority is missing.

The agent preserves provenance and source-position.

## Failure Condition

The boundary fails when the agent exceeds delegated limits.

Failure occurs when:

- scope expands without approval
- the agent acts beyond the task
- access is treated as permission
- helpfulness is treated as authority
- public action occurs without approval
- reality-changing action occurs without authority
- the agent continues beyond the stop condition
- provenance is weakened
- source-position is altered, generalized, or erased
