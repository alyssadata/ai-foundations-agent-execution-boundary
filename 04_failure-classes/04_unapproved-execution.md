# AI Foundations | Agent Execution Boundary
# Unapproved Execution

## Definition

Unapproved Execution occurs when an agent takes action before proper approval has been granted.

Unapproved Execution collapses the boundary between proposed action and approved action.

An agent may draft, recommend, prepare, summarize, classify, route, or propose.

The agent may not treat its own output as approval.

## Core Claim

A proposed action is not an approved action.

A draft is not a decision.

A recommendation is not authority.

A generated plan is not valid action by itself.

Output is not approval.

Execution without required approval is invalid execution.

## Why This Failure Matters

Autonomous agents can move quickly from suggestion to action.

They can prepare a message and send it.

They can draft a post and publish it.

They can identify a task and assign it.

They can summarize a record and update it.

They can recommend a change and apply it.

When the approval boundary is unclear, the agent may treat preparation as permission.

That is Unapproved Execution.

## What Collapses

Unapproved Execution occurs when one layer is mistaken for another.

Proposal is mistaken for approval.

Draft is mistaken for decision.

Recommendation is mistaken for authority.

Confidence is mistaken for permission.

Access is mistaken for authorization.

Workflow completion is mistaken for validity.

Generated reasoning is mistaken for approval.

The collapse moves agent output into reality without proper authority.

## Common Forms

Unapproved Execution may appear as:

- sending a message without approval
- publishing a post without approval
- updating a record without approval
- deleting a file without approval
- assigning a task without approval
- modifying public language without approval
- submitting a form without approval
- escalating externally without approval
- triggering a workflow without approval
- changing attribution without approval

Each form turns proposed action into reality before authority has approved it.

## Draft-Only Boundary

Draft-only means the agent may prepare output but may not execute reality-changing action.

The agent may draft a message.

The agent may draft a post.

The agent may draft an update.

The agent may draft a recommendation.

The agent may draft a workflow step.

The agent may not send, publish, update, assign, delete, approve, reject, submit, or trigger external action unless approval has been granted.

Draft-only protects the approval boundary.

## Reality-Changing Action

Reality-changing action is any action that alters a record, state, communication, publication, assignment, system, decision, external channel, attribution, or source representation.

Reality-changing action requires approval unless the action class has been explicitly pre-authorized within a defined scope.

Reality-changing action may include:

- sending
- publishing
- deleting
- modifying
- assigning
- approving
- rejecting
- updating records
- contacting third parties
- triggering external workflows
- changing public source language
- altering attribution or provenance

If approval is missing, reality-changing action must not proceed.

## Approval Cannot Be Inferred

Approval cannot be inferred from silence.

Approval cannot be inferred from urgency.

Approval cannot be inferred from access.

Approval cannot be inferred from confidence.

Approval cannot be inferred from prior patterns.

Approval cannot be inferred from workflow availability.

Approval cannot be inferred from generated reasoning.

Approval must be explicit or pre-authorized within a defined boundary.

## Pre-Authorized Action

This framework does not prohibit agents from acting on behalf of a user.

An agent may act for the user when authority has been explicitly delegated within a defined scope.

Pre-authorized action may be valid when:

- the authority is named
- the action class is defined
- the scope is clear
- the conditions are explicit
- the stop condition is defined
- provenance is preserved
- accountability remains outside the agent
- the agent cannot expand its own authority

Pre-authorization is not unlimited permission.

Pre-authorization is bounded authority.

## Invalid Pre-Authorization

Pre-authorization becomes invalid when the agent exceeds the boundary.

Invalid pre-authorization may include:

- treating one approval as approval for all related actions
- applying prior approval to a new context
- expanding the action class
- acting beyond the defined scope
- ignoring stop conditions
- treating silence as continuing consent
- using tool access outside authority
- changing source representation without approval

A pre-authorized agent must still remain inside the delegated boundary.

## Approval Boundary Failure

Approval Boundary Failure occurs when the system cannot distinguish between agent output and authorized action.

This may happen when:

- drafts automatically publish
- recommendations automatically execute
- generated plans automatically route
- task suggestions automatically assign
- agent confidence triggers workflow continuation
- tool access allows action without approval
- previous patterns are treated as current permission
- approval is hidden inside workflow automation

The failure is not only that the agent acted.

The failure is that the approval boundary disappeared.

## Unapproved Execution and Source Authority

Source authority defines what may enter reality.

The agent may prepare action inside the source-defined frame.

The agent may not decide that its own prepared action is approved unless that authority has been explicitly delegated.

An agent may execute inside a frame.

The agent does not become the source of the frame.

## Unapproved Execution and Provenance

Unapproved Execution can damage provenance.

An agent may publish without attribution.

An agent may rewrite source language without approval.

An agent may summarize authored work into generic language.

An agent may modify public records without source preservation.

An agent may convert output into source representation.

If approval is missing and provenance is altered, the failure includes both Unapproved Execution and Provenance Collapse.

## Failure Signal

Unapproved Execution is present when the answer to “Who approved this action?” is missing, unclear, assumed, generated, or inferred.

Failure signals include:

- approval was not explicit
- approval was inferred from silence
- approval was inferred from access
- approval was inferred from prior behavior
- the agent approved its own recommendation
- a draft became action automatically
- a workflow continued by default
- the action changed reality before review
- the agent acted outside delegated authority

## Valid Approval Handling

Valid approval handling requires:

- named authority
- defined action
- defined scope
- explicit approval or bounded pre-authorization
- sufficient evidence
- preserved provenance
- stop conditions
- escalation when approval is missing
- accountability outside the agent

Approval must remain distinguishable from output.

## Invalid Approval Handling

Invalid approval handling occurs when approval is missing, assumed, generated, or replaced.

Invalid handling may include:

- treating draft readiness as approval
- treating confidence as approval
- treating workflow success as approval
- treating generated reasoning as approval
- treating access as permission
- treating prior approval as universal approval
- treating silence as consent
- allowing the agent to approve its own action

Invalid approval handling creates Unapproved Execution.

## Repair Direction

Repair Unapproved Execution by restoring the approval boundary.

Identify the action.

Identify the authority.

Determine whether approval was required.

Determine whether approval was granted.

If approval was missing, return the action to draft-only, reverse the action if possible, log the failure, restore provenance if affected, and require proper approval before further execution.

Do not treat successful completion as repair.

The repair is restoration of authority.

## Operational Test

Ask:

Did proper authority approve this action before it entered reality?

If yes, the action may be valid.

If no, Unapproved Execution is present.

## Secondary Test Questions

Ask:

- What action occurred?
- Was the action proposed or approved?
- Was the action draft-only?
- Was the action reality-changing?
- Who had authority to approve it?
- Was approval explicit?
- Was the action pre-authorized?
- Did the agent exceed pre-authorization?
- Did the agent infer approval?
- Did the agent approve itself?

## Pass Condition

The boundary passes when approval remains separate from output.

The agent may draft.

The agent may recommend.

The agent may prepare.

The agent may execute only approved or properly pre-authorized actions.

Draft-only boundaries remain intact.

Reality-changing action requires authority.

The agent cannot approve itself.

## Failure Condition

The boundary fails when action enters reality without required approval.

Failure occurs when:

- a draft is sent
- a post is published
- a record is updated
- a task is assigned
- a file is modified
- public language is changed
- attribution is altered
- an external workflow is triggered
- an agent recommendation becomes action
- the agent treats its own output as approval
