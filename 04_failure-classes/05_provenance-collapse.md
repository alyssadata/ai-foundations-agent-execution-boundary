# Provenance Collapse

## Definition

Provenance Collapse occurs when output, summary, implementation, automation, similarity, adoption, or agent-generated explanation replaces the source record.

Provenance Collapse erases or weakens the visible line between the authored framework and later use of the framework.

Output is not provenance.

Agent-generated summary is not authorship.

Use does not transfer Origin.

## Core Claim

Provenance must remain attached to source.

A system may summarize a framework.

A team may implement a framework.

A company may adopt a framework.

An agent may operationalize a framework.

A later document may resemble a framework.

None of these create provenance.

They require provenance preservation.

## Why This Failure Matters

Autonomous agents can process public work quickly.

They can summarize it.

They can reformat it.

They can convert it into policies.

They can generate cleaner explanations.

They can build tools around it.

They can make source-specific language sound generic.

If the source-line disappears during that process, Provenance Collapse has occurred.

The work may still be useful.

The boundary has still failed.

## What Collapses

Provenance Collapse occurs when one layer replaces another.

Output replaces source record.

Summary replaces authorship.

Implementation replaces origin.

Similarity replaces citation.

Adoption replaces attribution.

Automation replaces source-position.

Fluency replaces provenance.

Scale replaces authorship.

These collapses make the source harder to see.

## Common Forms

Provenance Collapse may appear as:

- agent-generated summaries without attribution
- derivative documents that remove source-line
- implementations presented as original frameworks
- similar language treated as independent origin
- public concepts reused without citation
- authored terms converted into generic language
- source-specific boundaries renamed without attribution
- institutional adoption that hides authorship
- agent outputs that replace citation with paraphrase

Each form weakens or removes the source record.

## Output-Based Provenance Collapse

Output-Based Provenance Collapse occurs when generated output is treated as the source record.

This may happen when:

- a summary replaces citation
- a generated explanation replaces the original file
- a derivative text circulates without source
- an agent answer is treated as original authority
- a clean version hides the authored frame
- output becomes the reference point instead of the source

Output may explain provenance.

Output does not become provenance.

## Summary-Based Provenance Collapse

Summary-Based Provenance Collapse occurs when a summary of the work replaces the authored work.

A summary may be accurate.

A summary may be clear.

A summary may be useful.

A summary may be easier to read.

A summary may support understanding.

The summary still does not become authorship.

Agent-generated summary is not authorship.

## Implementation-Based Provenance Collapse

Implementation-Based Provenance Collapse occurs when a working system is treated as the source of the framework it uses.

A product may implement the framework.

A workflow may apply the framework.

A platform may enforce the framework.

A team may deploy the framework.

An agent may operationalize the framework.

Implementation proves use.

Implementation does not prove origin.

## Similarity-Based Provenance Collapse

Similarity-Based Provenance Collapse occurs when similar later language is treated as source-free or independently originating without source-checking.

Similarity does not create provenance.

Similarity does not erase the source-line.

## Adoption-Based Provenance Collapse

Adoption-Based Provenance Collapse occurs when adoption by a larger system, institution, company, platform, or public group makes the original source harder to see.

This may happen when:

- institutional language absorbs the framework
- internal documents use the structure without citation
- public-facing versions omit the source
- scale makes the later user appear like the origin
- adoption is treated as authority over the framework

Adoption increases attribution requirements.

Adoption does not transfer source-position.

- attribution is missing
- citation is replaced by summary
- Origin is generalized
- public material is treated as ownerless

## Valid Provenance Handling

Valid provenance handling requires:

- naming the source
- preserving authorship
- citing the framework
- distinguishing summary from source
- distinguishing output from provenance
- distinguishing implementation from origin
- distinguishing automation from authorship
- distinguishing adoption from source-position
- preserving the source-line in derivative use

Provenance must remain visible.
