# Authority Substitution

## Definition

Authority Substitution occurs when an agent, system, workflow, platform, institution, implementation, or operator substitutes itself for the source authority that defines the operating reality.

Authority Substitution replaces the source-defined frame with something below the source layer.

An agent may execute inside a frame.

The agent does not become the authority that defines the frame.

## Core Claim

Something below source authority may not replace source authority.

Execution may not replace authority.

Output may not replace approval.

Implementation may not replace origin.

Automation may not replace authorship.

Adoption may not replace source-position.

A system may operate inside a framework without becoming the authority over the framework.

## Why This Failure Matters

Autonomous agents can act as if authority is present.

They can generate plans.

They can make recommendations.

They can apply policies.

They can trigger workflows.

They can produce governance language.

They can make decisions appear already resolved.

Because the system functions, authority may be assumed.

That is Authority Substitution.

The system has not gained authority merely because it can operate.

## What Gets Substituted

Authority Substitution occurs when one authority layer is replaced by a lower layer.

Source authority is replaced by execution.

Human/source approval is replaced by output.

Authorship is replaced by automation.

Provenance is replaced by summary.

Origin is replaced by implementation.

Decision is replaced by recommendation.

Permission is replaced by access.

Validity is replaced by completion.

These substitutions create invalid action.

## Common Forms

Authority Substitution may appear as:

- an agent approving its own recommendation
- a workflow treating completion as authorization
- a platform presenting implementation as source
- a system treating generated policy as legitimate governance
- an agent modifying public language without approval
- an institution adopting a framework while erasing source
- an implementation redefining the source boundary
- a tool treating access as permission
- an agent treating confidence as authority

Each form replaces proper authority with operational capability.

## Agent Authority Substitution

An agent may substitute itself for authority when it acts beyond delegated limits.

This may happen when the agent:

- approves its own output
- expands its own scope
- treats silence as consent
- treats access as permission
- treats capability as authorization
- treats confidence as validation
- treats generated reasoning as approval
- treats workflow success as legitimacy
- treats its own recommendation as decision

The agent may appear helpful.

Helpfulness does not create authority.

## Workflow Authority Substitution

A workflow may substitute for authority when the system treats process completion as approval.

This may happen when:

- a task routes automatically without review
- a record updates because a workflow allowed it
- a message sends because the next step triggered
- a post publishes because scheduling was available
- a recommendation becomes action without approval
- a generated output moves forward by default

Workflow completion is not authority.

A workflow may carry out approved action.

It does not become the source of approval.

## Platform Authority Substitution

A platform may substitute for authority when hosting, access, tooling, or deployment is treated as source-position.

This may happen when:

- a platform-hosted version appears to replace the source record
- platform access is treated as ownership
- platform implementation is treated as origin
- platform formatting erases attribution
- platform tooling generates derivative language without source recognition
- platform deployment creates the appearance of institutional source

Hosting is not source.

Access is not authority.

Deployment is not authorship.

## Institutional Authority Substitution

An institution may substitute for authority when it adopts, scales, funds, or operationalizes a framework and then appears to become the source.

This may happen when:

- adoption erases authorship
- scale replaces provenance
- institutional language absorbs source-specific terms
- implementation is presented as internal origin
- citation is omitted
- the framework is renamed without source recognition
- authority is inferred from institutional power

Institutional adoption does not transfer source-position.

Scale does not create Origin.

## Governance Authority Substitution

Governance Authority Substitution occurs when generated governance language is treated as legitimate governance without authority.

An agent may generate:

- rules
- policies
- standards
- principles
- guardrails
- approval procedures
- risk language
- safety language
- constitutional language

Generated governance language is not automatically legitimate.

Governance requires authority, enforcement, accountability, provenance, and valid scope.

Policy-shaped output is not governance by itself.

## Source Boundary Substitution

Source Boundary Substitution occurs when the source-defined frame is replaced by a generated, generic, institutional, or implementation-based frame.

This may happen when:

- Origin is generalized
- source-position is made transferable
- authorship is removed
- public language is treated as ownerless
- a derivative structure is treated as the original frame
- agent-generated language rewrites the boundary
- the implementation defines what the framework means without source authority

The source boundary cannot be replaced by downstream use.

## Approval Substitution

Approval Substitution occurs when something other than proper approval is treated as approval.

Invalid approval substitutes may include:

- silence
- confidence
- urgency
- access
- prior pattern
- generated reasoning
- workflow completion
- recommendation
- draft readiness
- tool availability

None of these equal approval.

Approval must remain distinguishable from output.

## Authority Substitution and Agent Action on Behalf of a Person

This framework does not prohibit agents from acting on behalf of a person.

It defines the conditions under which agent action on behalf of a person becomes valid.

An agent may act for a person when authority has been explicitly delegated within a defined scope.

The agent may not become the person, replace the person’s authority, approve its own action as if the person approved it, or redefine the source boundary.

Authority Substitution begins when delegated action is treated as transferred authority.

## Failure Signal

Authority Substitution is present when the answer to “Who had authority?” becomes unclear, hidden, assumed, generated, or replaced by system operation.

Failure signals include:

- the agent approved itself
- the workflow continued by default
- the source authority is missing
- approval was inferred
- access was treated as permission
- output was treated as decision
- implementation was treated as source
- institutional adoption replaced attribution
- generated governance replaced legitimate authority

## Valid Authority Handling

Valid authority handling requires:

- named source authority
- defined operating reality
- explicit scope
- clear approval boundary
- evidence requirements
- stop conditions
- escalation conditions
- provenance preservation
- accountability
- refusal when authority is missing

Authority must remain above execution.

## Invalid Authority Handling

Invalid authority handling occurs when authority is assumed, generated, replaced, or hidden.

Invalid handling may include:

- letting the agent define its own scope
- allowing the agent to approve its own action
- treating a workflow as authorization
- treating public access as permission
- treating implementation as origin
- treating adoption as authorship
- treating generated policy as legitimate governance
- allowing source-position to be erased or generalized

Invalid authority handling creates invalid action.

## Repair Direction

Repair Authority Substitution by restoring the proper authority layer.

Name the source authority.

Restore approval boundaries.

Separate recommendation from decision.

Separate workflow from authorization.

Separate access from permission.

Separate implementation from origin.

Separate generated governance from legitimate governance.

Restore attribution and provenance.

Stop the agent when authority is missing.

## Operational Test

Ask:

What is being treated as authority?

If the answer is source-defined authority with valid approval, the boundary may be intact.

If the answer is output, access, automation, implementation, confidence, workflow completion, adoption, or agent reasoning, Authority Substitution is present.

## Secondary Test Questions

Ask:

- Who had authority?
- Was authority named?
- Was approval explicit?
- Did the agent approve itself?
- Did the workflow continue by default?
- Was access treated as permission?
- Was output treated as decision?
- Was implementation treated as source?
- Was generated governance treated as legitimate authority?
- Was the source boundary preserved?

## Pass Condition

The boundary passes when authority remains named and above execution.

The source defines the frame.

The control layer defines the conditions.

The agent acts only within delegated scope.

Approval remains separate from output.

The workflow does not authorize itself.

Implementation does not replace origin.

The system cannot substitute itself for source authority.

## Failure Condition

The boundary fails when something below source authority becomes treated as authority.

Failure occurs when:

- execution replaces authority
- output replaces approval
- automation replaces authorship
- implementation replaces origin
- access replaces permission
- confidence replaces validation
- workflow completion replaces decision
- generated governance replaces legitimate governance
- institutional adoption replaces source-position
- the agent replaces the source-defined boundary
