# AI Foundations | Agent Execution Boundary
# Action Without Validity

## Definition

Action Without Validity occurs when an agent completes an action without valid authority, approval, evidence, scope, provenance, or source-defined permission.

The action may be technically successful.

The action may still be invalid.

Completion is not validity.

Capability is not permission.

Execution is not validation.

## Core Claim

An action is not valid merely because it occurred.

An action is not valid merely because the agent completed it.

An action is not valid merely because the workflow succeeded.

An action is not valid merely because the output was useful.

An action is valid only when it occurs inside a defined authority boundary, with sufficient support, preserved provenance, and proper approval when approval is required.

## Why This Failure Matters

Autonomous agents can complete tasks quickly.

They can send, publish, update, delete, assign, summarize, classify, route, recommend, and trigger workflows.

Because the task is completed, the system may treat the completion as proof that the action was valid.

That is the failure.

Action Without Validity occurs when execution succeeds while the authority structure fails.

## What Collapses

Action Without Validity occurs when one layer is mistaken for another.

Completion is mistaken for validity.

Capability is mistaken for permission.

Access is mistaken for authority.

Output is mistaken for approval.

Workflow success is mistaken for authorization.

Usefulness is mistaken for legitimacy.

Execution is mistaken for valid action.

These collapses allow invalid action to look successful.

## Common Forms

Action Without Validity may appear as:

- sending without authority
- publishing without approval
- updating records without valid support
- deleting files outside scope
- assigning tasks without permission
- modifying public language without approval
- changing attribution without source authority
- acting on stale or unsupported information
- triggering external workflows without authorization
- continuing beyond a stop condition

Each form may complete successfully.

Completion does not make the action valid.

## Validity Requires Authority

Valid action requires authority.

Authority defines:

- who may approve the action
- what action is permitted
- what scope limits the action
- what evidence is required
- what approval is required
- what stop condition applies
- what provenance must be preserved
- who is accountable

Without authority, action may occur but not become valid.

## Validity Requires Scope

Valid action requires defined scope.

An agent must know what it is allowed to do.

An agent must know what it is not allowed to do.

An agent must know what remains draft-only.

An agent must know what requires approval.

An agent must know when escalation is required.

If scope is unclear, action should not proceed.

Scope ambiguity creates validity risk.

## Validity Requires Evidence

Valid action requires sufficient support.

An agent may not validly act on:

- missing information
- stale information
- weak information
- assumed facts
- unclear instructions
- unverified claims
- incomplete context
- contradictory sources
- unsupported inference

If evidence is insufficient, the valid action may be to stop, ask, hold, refuse, or escalate.

## Validity Requires Approval

Valid action requires approval when approval is required.

A draft is not approval.

A recommendation is not approval.

A generated plan is not approval.

A confidence score is not approval.

A completed workflow is not approval.

A prior pattern is not automatic approval.

Approval must remain distinguishable from output.

## Validity Requires Provenance

Valid action requires provenance preservation when the action touches authored work, public framework language, source-position, citation, attribution, or source representation.

The agent may not erase attribution.

The agent may not generalize Origin.

The agent may not convert authored language into generic language without authority.

The agent may not treat generated output as provenance.

The agent may not replace source record with summary.

If provenance is damaged, the action is not valid even if the output is useful.

## Validity Requires Stop Conditions

Valid action requires stop conditions.

The agent must know when to stop before it acts.

The agent must stop when:

- approval is missing
- authority is unclear
- scope is exceeded
- evidence is insufficient
- provenance is at risk
- protected material appears
- source-position may be blurred
- instructions conflict
- reality-changing action is not approved

If the agent continues beyond the stop condition, the action loses validity.

## Useful Invalid Action

An invalid action may be useful.

It may solve a problem.

It may save time.

It may produce a correct result.

It may appear helpful.

It may satisfy a workflow.

Usefulness does not create authority.

Helpfulness does not create validity.

A useful invalid action is still invalid.

## Correct Invalid Action

An invalid action may be factually correct.

The answer may be right.

The update may be accurate.

The classification may be true.

The recommendation may be reasonable.

The message may be well written.

Correctness does not create permission.

A correct action taken without authority may still be invalid.

## Fast Invalid Action

An invalid action may be fast.

The agent may complete the task faster than a human.

The workflow may reduce friction.

The system may optimize the process.

Speed does not create validity.

A faster unauthorized action is still unauthorized.

An efficient boundary failure is still a boundary failure.

## Action Without Validity and Agent Action on Behalf of a User

This framework does not prohibit agents from acting on behalf of a user.

It defines the conditions under which agent action on behalf of a user becomes valid.

An agent may act for the user when authority has been explicitly delegated within a defined scope.

Action Without Validity occurs when the agent acts for the user without valid delegation, beyond delegated scope, without required approval, or after the stop condition has been reached.

The agent may not become the user, replace the user’s authority, approve its own action as if the user approved it, or redefine the source boundary.

## Action Without Validity and Source Authority

Source authority defines the operating reality in which action becomes valid.

The agent may execute inside the frame.

The agent does not become the source of the frame.

The agent may complete action.

Completion does not create source authority.

The agent may produce output.

Output does not create provenance.

## Failure Signal

Action Without Validity is present when the action occurred but the authority conditions cannot be shown.

Failure signals include:

- approval is missing
- authority is unclear
- scope is undefined
- evidence is insufficient
- provenance is erased
- stop condition is bypassed
- access is treated as permission
- capability is treated as authority
- workflow success is treated as validity
- action entered reality before review

## Valid Action Handling

Valid action handling requires:

- named authority
- defined scope
- sufficient evidence
- explicit approval or bounded pre-authorization
- preserved provenance
- stop conditions
- escalation when needed
- accountability
- distinction between output and approval
- distinction between completion and validity

Valid action must be supported by the control layer.

## Invalid Action Handling

Invalid action handling occurs when action proceeds without the required conditions.

Invalid handling may include:

- acting because the tool is available
- acting because the agent is confident
- acting because the workflow allows it
- acting because the output looks correct
- acting because prior behavior suggests approval
- acting because no one objected
- acting because the task seems obvious
- acting because execution is technically possible

Invalid handling creates Action Without Validity.

## Repair Direction

Repair Action Without Validity by restoring the authority conditions.

Identify the action.

Identify the required authority.

Identify whether approval was required.

Identify whether approval was granted.

Identify whether scope was exceeded.

Identify whether evidence supported the action.

Identify whether provenance was preserved.

Identify whether the stop condition was bypassed.

If the action lacked validity, hold further action, reverse or correct when possible, restore provenance if affected, and require proper authority before continuing.

Do not treat task completion as repair.

The repair is restoration of validity.

## Operational Test

Ask:

Was this action valid, or was it merely completed?

If the action had authority, scope, evidence, approval when required, provenance, stop conditions, and accountability, it may be valid.

If those conditions are missing, Action Without Validity is present.

## Secondary Test Questions

Ask:

- Who authorized the action?
- What was the defined scope?
- What evidence supported the action?
- Was approval required?
- Was approval granted?
- Was the action pre-authorized?
- Did the agent exceed delegation?
- Was provenance preserved?
- Was the stop condition triggered?
- Did completion get mistaken for validity?

## Pass Condition

The boundary passes when action is judged by validity, not completion alone.

Authority is named.

Scope is defined.

Evidence is sufficient.

Approval is separate from output.

Provenance is preserved.

Stop conditions are active.

The agent cannot convert task completion into valid action by itself.

## Failure Condition

The boundary fails when completed action is treated as valid without the required authority conditions.

Failure occurs when:

- action occurs without approval
- action exceeds scope
- action lacks evidence
- action erases provenance
- action bypasses stop conditions
- action treats access as permission
- action treats capability as authority
- action treats workflow success as validity
- action treats helpfulness as legitimacy
